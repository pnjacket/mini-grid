# doc-excavate

Operationalizes the standard's **reverse-authoring (code→doc bootstrap)** flow (STANDARD Part 10f) — the brownfield counterpart to `doc-scaffold`. Advisory: confirm before writing files; never overwrite without asking. **`doc-scaffold` is deliberately code-blind** (greenfield: never assume the product from the repo); this skill is the opposite — the **code is the primary source of truth**, and the interview fills only what code cannot prove.

Its job is a **faithful, drift-clean baseline**, not a finished spec: extraction lands most concerns at **Sketch/Specified** with a structural Contract-grade surface, and it hands off (to `doc-levelup`/`concern-specialist`) to raise the rest. It does **not** make the doc set build-ready on its own.

## Inputs you rely on
- The Dictum standard: `STANDARD.md` (esp. **Part 10f**), `concerns/11.x-*.md`, `templates/` (skeletons — manifest, concern-doc, binding-map).
- The **`code-cartographer` agent** — the read-only sweep that produces the trait read, candidate contract inventory, draft binding map, verification state, recoverability report, and smell list. Run it first; this skill authors from its output.

## Procedure

1. **Confirm the target and the read-only stance.** Get the target repo path. State plainly: this skill **reads the product code but never edits, commits, or pushes it** — it writes only the doc set (manifest, concern docs, `bindings.yaml`, index). Confirm where the generated docs go (the target's working tree by default; a separate output dir if the operator wants the product left pristine).
2. **Run `code-cartographer`** over the repo. Take its inventory, trait read, draft bindings, recoverability report, and smell list as your raw material. Everything it produced is a **candidate**, not a settled fact.
3. **Derive breadth from the observed traits** (the code accelerates this vs `doc-scaffold`'s blank start): always the 6 Core + 2 Baseline; add each Module concern whose trait *fired in the code*. **But do not read code silence as scope-out** — a trait absent from code (`commercial`, `regulated`) is *unknown*, an interview question, not an `absent`. Present the derived scope for the operator to confirm/adjust.
4. **Confirm-and-fill interview — to saturation** (the interview-first rule, Part 0.6, adapted: the code gives a draft to *react to*, which is a higher-quality interview than blank-page generation). Run in small rounds, driven by a coverage map, until a round surfaces nothing new or the operator calls it — **never stop because you've "inferred enough."** Two tracks:
   - **Confirm track** — replay the cartographer's inferences as `[ASSUMPTION]` markers for the operator to **confirm or correct**: capability boundaries (`CAP-###`, always low-confidence), trait values, entity/endpoint intent, which smells are real. Confirmed → clear the marker; uncorrected → the `[ASSUMPTION]` **stays in the published doc** (Part 6), visible for audit.
   - **Fill track** — the **must-interview gaps** the recoverability report named, which code *cannot* answer: requirements intent / personas / success metrics; **Non-goals + each scope-out's kind** (`absent` vs `deferred`, Part 9 — categorically un-extractable: code can't tell "deliberately excluded" from "not built yet"); the security threat model & policy intent; performance/SLA targets; governance/compliance context; business/legal model. Unanswered → `[GAP]`/Open Question, never a guess.
   - **Aspirational-contract track (the dominant brownfield decision).** For **each scaffolded-but-unbuilt contract** the cartographer flagged (orphan table, README-claimed-but-unimplemented feature, dangling client call), ask the operator which it is — this *is* the concrete form of the Part 9 excluded-vs-not-built ambiguity: **`deferred`** (planned; record the contract as `[FUTURE-SCOPE]`/scoped-out `deferred` with a re-entry note, **not** a delivered contract), **`absent`** (abandoned / never intended; drop it, noting the dead structure as a `[REVISIT]` for the code owner), or a **defect** (the behavior is *owed*; keep the contract but mark the missing realization a `[GAP]`). **Never mint an aspirational contract as delivered or Verified** — its structure existing in schema/doc/UI is not its behavior existing in code.
5. **Write the manifest** from `templates/manifest.template.md`: traits, packaging, per-concern `in_scope` + **honest** `current_rung` (what is *actually* authored after this pass — usually sketch/specified, not contract-grade) + `target_rung` (default contract-grade) + `out_of_scope_subaspects` (published keys, each with its `absent`/`deferred` kind). Add a **provenance** note — reverse-authored from code at `<base ref/commit>` — so consumers know this set was bootstrapped, not authored doc-first. **In autonomous mode the scope-out *kind itself* is an inference** (code can't tell `absent` from `deferred`, Part 9): mark the absent-vs-deferred decision `[ASSUMPTION]`, not just the intent behind it.
6. **Generate the doc set** from the templates, populated from the confirmed inventory + interview. This is an **extract → synthesize** step, not a transcription: the cartographer's inventory is fine-grained (code grain); author a **navigable hierarchy**, not a flat list.
   - **Granularity is controlled, default `fine`.** Keep contracts at the fine code grain (one per endpoint/entity/component — maximal explicit context for an AI implementer, each independently drift-checkable) and **synthesize a coarse index over them**: `CAP`/resource groupings wired to their constituents via the cartographer's reference web (`CAP→API/SCREEN`, `SCREEN→API`, `API→ENTITY`). Honor a manifest `authoring_granularity` (`fine`|`grouped`, default `fine`, optional per-concern) for the *primary* level; author **owned-once at each level** and cross-linked, so both views resolve and drift/reconciliation can key on either. **Do not roll fine contracts up to guess an author's coarser grain** — that grain is intent you can't recover; the fix for a flat extraction is *linking + a coarse index*, never fewer contracts.
   - **Document as-built:** own every contract once by stable ID, reference across concerns by ID (never duplicate), **wire the reference web** (the edges are most of the map's value — a node dump with no edges is the real failure). Mark every unconfirmed inference `[ASSUMPTION]` and every smell `[REVISIT]`/Open Question — **do not enshrine a suspected defect as a contract, and do not silently "fix" it.** Leave `[GAP]` where the interview couldn't fill.
7. **Mint `bindings.yaml`** from the cartographer's draft — the brownfield byproduct that immediately unlocks `drift-detector` + `implementation-planner`. Finalize it to `templates/binding-map.template.md` rules: keys are confirmed in-scope IDs; **own-code locators only** (substitution rule); declare `coverage:` (`fully_bound` vs `curated` with a stated locus); attach existing tests as `asserted_by` where present. **An aspirational contract kept as a `deferred`/`[GAP]` behavior gets no behavioral binding** — it is a coverage gap by construction (that is precisely what the drift-detector/planner will read as "build this"); do not fabricate a locator to make the map look complete. **Promote every ID you mint to a bindable entry** — including IDs minted *inline* in prose/tables (a route annotation inside a nav table, a derived-key mention): a code-realizable ID that exists only in prose reads as an unbound coverage gap later, so give it a binding row (or fold it into its owning contract) now. **Use the test's verbatim symbol/title** in `asserted_by` (not a regex approximation) and confirm each resolves before writing it. Record existing passing tests as **candidate Verified evidence for Delivery** — do **not** set the Verified rung yourself. This baseline adopts code-as-built into the docs, resolving the initial "everything is code-ahead" drift into a fixpoint.
8. **Generate the human index/README** from the manifest (derived; regenerate whenever the manifest changes — the auditor flags README↔manifest drift).
   - **If a prior doc web already exists (re-adoption, not first bootstrap): reconcile, don't overwrite.** An excavation coins a **new candidate ID space** and tends to **over-fragment** (one prior contract surfacing as several finer ones), so it is never a drop-in replacement. Run the Part 10f **ID-reconciliation**: join excavated IDs to the incumbent IDs (verbatim where in-code `// DICT: <ID>` annotations survived, by body-comparison where not), write a **tombstone-alias table** (Part 10d — never silently reuse or drop an incumbent ID), **roll up** over-fragmented contracts to the incumbent granularity, and **flag every unmatched contract** for the operator. Onto an empty web (first bootstrap) there is nothing to reconcile.
9. **Report and hand off.** Produce: an **assumption/gap register** (every `[ASSUMPTION]`/`[GAP]`/`[REVISIT]` left open), an **honest rung table** (per concern actual rung — expect sketch/specified), and next steps — run `doc-maturity-auditor` (confirm rungs + cross-refs), then `drift-detector` (now possible: prove the docs match the code they came from), then `doc-levelup`/`concern-specialist` on the must-interview concerns to reach build-ready.

## Guardrails
- **Read the product; never edit/commit/push it.** Write only the doc set (docs + manifest + bindings + index), into the working tree (or a separate output dir the operator names). Nothing is committed by this skill.
- **Extract structure; interview intent.** Code proves fields/routes/constraints; it never proves *why*, *what was left out*, the threat model, SLAs, or the business model — those are interview questions, and unanswered ones are `[GAP]`, never guesses. Code silence ≠ out of scope.
- **Every code-inferred fact is `[ASSUMPTION]` until confirmed;** uncorrected ones stay in the published doc for audit (this is the operator-present form of the autonomous concession, Part 0.6).
- **As-built, not enshrined.** Suspected defects/contradictions/dead code are `[REVISIT]`, not contracts.
- **Honest rungs; never claim Verified.** Report actual section-completeness; existing passing tests are candidate evidence for Delivery, not a rung you assign.
- **Interview to saturation,** not "inferred enough" — an unasked in-scope question is a silent assumption, the exact failure the standard exists to prevent.
- **Never weaken Contract-grade;** right-size via scope only. Reference the standard by bare name (resolved by the target's install pointer).
