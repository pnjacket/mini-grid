# The Documentation Standard

> A product-agnostic specification for what a *good software documentation set* must contain, so that producing build-ready docs is a structured, repeatable **method** for any product — from a 200-line single-purpose CLI to a multi-product SaaS platform with a portal.
>
> This document is **self-exemplifying**: it obeys its own document contract. Build-phase notes are in `<!-- BUILD: ... -->` comments and vanish on publish; `[GAP]`/`[FUTURE-SCOPE]`/`[REVISIT]` are subject markers about this standard's own state.

---

## Part 0 — Philosophy & First Principles

1. **Repeatability is about IMPLEMENTATION, not documentation.** A product's doc set is authored ~once and then *enhanced* over its life; it is never re-derived from scratch. The method must apply reliably across *different* products and produce consistent *structure* — not identical output for the same product twice. (That lifecycle — *how a living doc set is enhanced* — is **Part 10d**.)
2. **AI and human consumers are co-equal.** The standard demands enough explicitness for an AI to build without guessing, while remaining readable and navigable for people.
3. **Build-ready means a contract an implementer can satisfy without asking questions.** This is the entire purpose; every rule below serves it.
4. **One mechanism scales the whole spectrum: scope.** Breadth of scope (which concerns and sub-aspects apply) covers CLI↔platform *and* PoC↔production. There is no separate "rigor" or "intent" axis.
5. **Owned once, referenced everywhere.** Every contract has exactly one owning concern; others reference it by stable ID. This is the structural defense against the layer-drift failures the method exists to prevent.
6. **Dictum is interview-first; inference is an autonomous-mode concession, not the norm.** A doc set is build-ready (0.3) by asking the *operator* the questions an implementer would otherwise have to ask — paid at authoring time, on purpose. **Two elicitation modes, and the distinction is load-bearing:**
   - **Interactive (the intended default — an operator is present).** Intake and authoring drive to **requirement saturation**: many small rounds of questions, continuing until a round surfaces nothing new *or* the operator calls it. The tooling does **not** infer-and-proceed from a thin brief. **Adaptiveness (the "not a fixed questionnaire" rule) governs *which* questions are in scope (Part 9), never *how few* are asked** — adaptive ≠ thin. An unasked in-scope question here is a silent assumption and a defect.
   - **Autonomous (unattended — no human to interview, e.g. the standard's own validation builds).** This is the *only* mode in which the tooling substitutes inference for a question, as the deliberate price of running without an operator — and then **every inferred answer is recorded as an `[ASSUMPTION]` marker (Part 6) and surfaced for audit**, precisely because no one was asked. Inference is never the intended path; it is autonomy's cost, made auditable.

   The historical infer-heaviness of Dictum's validation drives is a fact about *those unattended runs*, not about the standard: with an operator in the room, Dictum interviews.

## Part 1 — The Model at a Glance

The system is **layered**: this **Standard** → **Templates** (fill-in skeletons) → a generative **Playbook**, operationalized by Claude Code tooling (Part 10b).

A doc set is described by **two structural axes**:

| Axis | Question | Mechanism |
|------|----------|-----------|
| **Breadth** | *Which* concerns and sub-aspects are in scope? | Scope, derived by an adaptive intake interview → modules |
| **Depth** | *How complete* is each concern's doc? | A universal 5-rung maturity ladder |

## Part 2 — Breadth: The Concern Catalog

Fifteen concerns in three behaviors. **Core** = always present, substantive. **Baseline** = always present, minimal by default, depth scales with risk. **Module** = absent unless a trait switches it on.

| # | Concern | Behavior | Trigger |
|---|---------|----------|---------|
| 11.1 | Product & Requirements | Core | always |
| 11.2 | Domain & Data | Core | always |
| 11.3 | Architecture | Core | always |
| 11.4 | Interfaces & Contracts *(sub-typed: CLI/HTTP/library/events/UI)* | Core | always |
| 11.5 | Quality & Testing | Core | always |
| 11.6 | Delivery Process | Core | always |
| 11.7 | Security & Privacy | Baseline | always; risk raises depth |
| 11.8 | Governance & Compliance | Baseline | always; regulation raises depth |
| 11.9 | User Experience | Module | interactive UI |
| 11.10 | Operations & Infrastructure | Module | deployed running service |
| 11.11 | Observability & Monitoring | Module | deployed running service |
| 11.12 | Integrations & External Dependencies | Module | third-party deps |
| 11.13 | Performance & Scalability | Module | real perf/scale needs |
| 11.14 | Accessibility & Internationalization | Module | a11y: any UI · i18n: multi-locale |
| 11.15 | Business & Legal | Module | commercial/distributed |

**Scope operates at two granularities:** concern-level (whole concern in/out) and **sub-aspect-level** (within a concern — e.g. error-handling, edge-cases, failure-modes in/out). See Part 9. Full per-concern specifications are in `concerns/` (Part 11).

## Part 3 — Depth: The Maturity Ladder

One universal 5-rung ladder, applied to every concern (each concern defines what each rung *means* for it):

```
Absent → Sketch → Specified → Contract-grade │ Verified
└──────── owned by the doc system ───────────┘  └─ owned by Delivery Process
```

- **Absent** — not addressed. **Sketch** — intent captured. **Specified** — complete prose requirements; a human could build with judgment. **Contract-grade** — testable acceptance + explicit contracts; **an AI can build without guessing** (the author's finish line).
- **Verified** — built and proven by a real-flow test. Same vocabulary, but owned/tracked by Delivery Process, not by doc authoring. **Doc maturity ≠ implementation status.**

Depth is per-concern: concerns may sit at different rungs at the same time.

## Part 4 — The Document Contract

Every doc shares one skeleton, and *which sections are complete IS its maturity*:

| Section | Appears at |
|---|---|
| Purpose & scope | Sketch |
| Non-goals / Out-of-scope | Sketch *(load-bearing — Part 9)* |
| Requirements (outline → complete) | Sketch → Specified |
| Open Questions | Sketch onward |
| Dependencies & Cross-references | Specified |
| Examples / Worked scenarios | Specified |
| Design Decisions | Specified → Contract-grade |
| Contracts | Contract-grade |
| Acceptance criteria *(testable & unambiguous — each maps to an observable check)* | Contract-grade |
| Status markers | all rungs |

**Contracts section, shape-adaptive.** For most concerns *Contracts* lists object-like contracts (entities, endpoints) by ID. For **process or policy concerns** whose owned contracts are inherently table- or prose-shaped and already fully stated in *Requirements* (e.g. Delivery's slice rule/DoD/playbook, Quality's coverage map, Governance's `POLICY-###`, Observability's SLO/alert tables, Integrations' `DEP-###` register), *Contracts* may be a **pointer/recap** to those Requirements rather than a re-listing — completeness is judged by the bar, not by a duplicate section. Do not invent a spurious ID prefix (e.g. `SLICE-###`) just to populate it. *(In practice, several concerns independently hit the template-vs-house-style mismatch here.)*

## Part 5 — Contracts Between Concerns

- **Owned once, referenced everywhere.** Each contract lives in its owning concern; other docs link to it.
- **Stable concern/contract IDs** independent of file layout, so merge/split (Part 8) never breaks a reference.
- The cross-concern **ID registry** (`CAP-### · ENTITY-### · INV-### · COMPONENT-### · API/CLI/LIB/EVT-### · ROLE-### · POLICY-### · SCREEN-### · ENV-### · DEP-### · PERF-### · LICENSE-TIER-### · SUCCESS-### · PERSONA-### · ADR-### · …`) is defined across the Part 11 specs and forms a single traceability web. A doc set may coin additional **local prefixes**; an ID *resolves* if it is consistently defined and referenced within the set, so an off-registry prefix is not itself a finding. Numeric `###` is the **default** suffix form; **semantic suffixes are an allowed variant** of the same scheme (`CAP-003` and `CAP-MODEL-CREATE` are equally stable IDs) — what binds is stability and uniqueness, not the digit form.
- **Forward references ("described-here, minted-by-X").** Dependency-ordered authoring (Part 10) still leaves seams where a consumer doc needs a contract its producer has not yet minted. The consumer must **not** mint the ID itself (owned-once); it records a placeholder — the binding *described here*, the ID *minted by* the named owning concern — and the reference is **tightened to the real ID when the producer reaches Contract-grade**. Tightening a by-description binding in an already-Contract-grade consumer is the sanctioned authoring-order seam, not drift.
- **Contracts-as-bounds.** A contract may **bound a value it deliberately delegates without fixing it** — a minimum contrast ratio over design-delegated color tokens, a frame budget over implementation-chosen algorithms. The bound is the owned, testable contract; the value inside it remains the delegate's freedom. The pattern recurs wherever accessibility/performance constrain design-delegated choices.

## Part 6 — Markers, Draft/Published & Provenance

- **Subject markers (durable, stay in published docs):** `[GAP]` `[ASSUMPTION]` `[REVISIT]` `[FUTURE-SCOPE]` — about the *product*. `[GAP]` is strictly about the product, never about doc-maturity.
- **Build markers (draft-only, stripped on publish):** document-maturation scaffolding / next-rung targets / author TODOs, written as `<!-- BUILD: ... -->` HTML comments. They never render and are noise during implementation.
- **Draft ↔ Published:** front-matter `status` flag; a **publish step** strips build markers and flips the status.
- **Original preservation:** git history + a version bump on publish. No parallel copies.

## Part 7 — The Product-Profile Manifest

A single machine-readable manifest is the source of truth: traits, in-scope concerns (+ sub-aspects), concern locations (by stable ID), current rungs, status. The human-facing index/README is **derived** from it — **generated at scaffold time** (Part 10b) and thereafter kept consistent: the `doc-maturity-auditor` checks that the index agrees with the manifest (same in-scope concerns + rungs) and flags drift, so a hand-edit that diverges is caught rather than silently trusted. The manifest, not the README, remains authoritative.

## Part 8 — Packaging

- **Operator's call, guided by heuristics.** One-file (concerns as sections) for small products; one-file-per-concern for large. Split a concern out when it outgrows ~a page or gains distinct owners.
- Stable-ID addressing (Part 5) makes any repackaging safe.

## Part 9 — Scope as the Only Calibration Lever

- The whole CLI↔platform **and** PoC↔production spectrum is handled by **scope** alone — there is no rigor/intent axis.
- **PoC** = narrower scope: concerns *and* sub-aspects marked out-of-scope (no compliance, no ops, happy-path only). **Production** = broader scope.
- **Contract-grade is one fixed, absolute bar** — applied to a smaller surface for a PoC, never a weaker bar.
- The **Non-goals / Out-of-scope** section (Part 4) records this calibration.
- **Risk/scope mismatch** (e.g. real PII present but its protection sub-aspects scoped out; or an interactive UI present with Accessibility & i18n (11.14) entirely scoped out) → advisory **warning only**; nothing blocks.
- **Scope-out has two kinds, and both are recorded.** `absent` — the subject **doesn't exist in the product** (no event bus, no auth surface, no third-party deps): the published key still goes in `out_of_scope_subaspects`, with the **trait fact as its justification**, and **no `[FUTURE-SCOPE]` re-entry note is owed** — there is nothing to re-enter. `deferred` — the subject exists but is calibrated out of this effort: same recording, **plus a re-entry note owed** in Non-goals. The justification line is what distinguishes them; conflating the kinds either litters absent subjects with re-entry notes or silently drops deferred ones.
- **A scope-out in one concern can punch a *traced* hole in another concern's bar.** A conformance claim whose criteria have preconditions another concern scoped out (e.g. an accessibility level containing responsive-layout criteria while layout adaptation is out of scope) is stated as **the claim minus traced exceptions** — each exception naming the scope decision it traces to — never silently narrowed, and never read as making the claiming concern's bar unsatisfiable.
- **Sub-aspect keys are a published vocabulary, not derived.** Each concern spec's *Sub-aspects* table publishes a **`key`** column — the canonical, stable identifier for each sub-aspect. A manifest's `out_of_scope_subaspects` entries must be exactly those published keys of the owning concern (e.g. `migrations-versioning`, `authorization`, `encryption`), so every scope decision is checkable against a known vocabulary rather than an ad-hoc string. An entry that is **not** a published key of its concern is a build error; a near-miss that still uniquely matches one row (e.g. a key from an older derivation rule, like `migrations-and-versioning`) is a **warning**, not a build error. *(The published keys follow a normalization — lowercase; drop everything from the first `:`, `(`, or `,`; strip markdown backticks and `-###` ID hints; collapse `/ + & ↔` and whitespace to single hyphens — but consumers reference the published column, never re-derive it.)*
- **Fully-scoped-out concern.** A concern with *all* its testable sub-aspects scoped out reaches Contract-grade once its residual scope is **fully asserted**: (a) the in-scope baseline facts are stated **and** (b) each scoped-out sub-aspect is listed in Non-goals with a one-line justification. There is then no Contracts/Acceptance section to complete because nothing testable remains in scope; the auditor must not flag this as an inflated rung. (A sub-aspect is **testable** if it owns or contributes a checkable contract — `ENTITY/INV/API/POLICY/…`; a purely factual residual one, e.g. "trust boundary = a single local user", is not.)
- **Mostly-scoped-out concern — the residual-scope Contract-grade checklist.** A module concern whose *bar* (in its `concerns/11.x` spec) reads as if the whole feature is present can still be Contract-grade on a narrow in-scope surface. The bar applies **to the in-scope surface only**; "Contract-grade" is reached by fully asserting the residual, via this checklist: **(1)** every in-scope fact is stated precisely; **(2)** every scoped-out **published sub-aspect key** is listed in Non-goals with a one-line justification; **(3)** those keys match the manifest's `out_of_scope_subaspects` exactly; **(4)** each in-scope assertion still owed a contract has an observable acceptance check. A concern spec's bar must not be read as unsatisfiable just because a sub-aspect it names (e.g. pricing, business-SLAs) is out of scope. *(In practice, Business & Legal 11.15 reaches Contract-grade with pricing/SLAs scoped out by deriving exactly this checklist — it should be prescribed, not re-derived each time.)*
- **Scope is decided at sub-aspect-key granularity.** The in/out decision is made on a concern's **published `key`** (the canonical vocabulary above). Finer facets *within* a key are prose under that key, not separate scope entries — so a key with no finer split, or one key that maps to **several owned contracts** (e.g. one `a11y-conformance` key behind four `A11Y-###` contracts; one `data-handling-policy` key behind several `POLICY-###`), is normal and not a defect. Only the published key appears in `out_of_scope_subaspects`. *(In practice, several concerns have coarser keys than their owned-contract set; this is expected.)*

## Part 10 — The Generative Playbook & Build-Ready Gate

**Phases (guided, ordered; flexible/iterative within each):** Intake → Scaffold → Author-to-rung → Review → Publish.

- **Intake** is an *adaptive*, **iterative-to-saturation** LLM interview (not a fixed questionnaire) that converges on the manifest (Part 7). Adaptive ≠ thin: relevance is scoped to the product's traits (Part 9), but within that scope the interview runs in **rounds** — small batches of questions, driven by a **coverage map** (each in-scope concern's sub-aspects × each capability's lifecycle, edge, failure, and non-functional dimensions) — and continues until a round surfaces nothing new or the operator stops it. The same elicitation loop runs in **Author-to-rung** (per concern, per feature), not just at intake. Stopping because the model has "inferred enough" is the prohibited mode in interactive use (Part 0.6): an unasked in-scope question becomes a silent assumption, which is exactly the failure the standard exists to prevent. **Unattended** (the autonomous-mode concession of Part 0.6) is the only time the interview degrades to inference — and then every inferred answer is marked `[ASSUMPTION]` and listed for later operator review, never silently settled.
- **Authoring order:** dependency-ordered — Product & Requirements → Domain & Data / Architecture → Interfaces → downstream — so owned contracts exist before downstream docs reference them.
- **Build-ready gate (handoff to implementation):** every **in-scope** concern has reached **Contract-grade**. Deferral is via scope (Part 9), not partial rungs.
- **After first publish**, the doc set enters the **Enhancement Lifecycle** (Part 10d) for the rest of the product's life — authored once, enhanced continuously (Part 0.1).

### Part 10a — The Fidelity-Staged Definition of Done

"Done" is staged by environment fidelity, because a developer cannot validate against infrastructure they don't control:

- **Merge gate** (developer stage) — the bar for code entering the repo. Validated at the highest local fidelity, with declared **external** dependencies substituted per the **environment fidelity map** (owned by Operations & Infrastructure, `ENV-###`).
- **Release gate** (real-infra, DevOps) — full-fidelity validation. Real-infra checks not runnable at merge are **deferred to the release gate and tracked, never silently skipped**.
- **The substitution rule:** the system's **own** code paths are always real (never bypass own auth/logic/nav); declared **external** dependencies may be substituted by approved fakes. Mocking S3 is legitimate; bypassing login is never. This is the environment-relative meaning of "real flow" in the `E2E-STANDARD` (Quality & Testing).
  - **Fidelity may differ by test tier *within* one environment**, and the **gate's E2E fidelity is what binds**: e.g. a real S3 container drives the merge-gate E2E while unit tests mock S3 — both legitimate; the gate's E2E is the fidelity that counts. The `ENV-###` map records this, and may carry an explicit checkable **`auth: real, no bypass`** assertion per environment so the headline rule is a map property, not just prose.
  - **The external may *be* the substitute.** A self-hosted stand-in (Mailpit for SMTP, MinIO for S3, a local Keycloak) run **real in every environment** satisfies the rule; the vendor swap is `[FUTURE-SCOPE]` behind the same `DEP-###` contract. (The "real at prod / faked at dev" framing is one case, not the only one.)
  - **The platform/host is itself a fidelity dimension.** What the product *runs on* — runtime, GPU/renderer, browser build — can be the substituted thing (software-GL standing in for a hardware GPU; a bundled engine for a branded browser); the `ENV-###` map records it per environment alongside per-external fidelity (Ops 11.10).
  - **The input-device/browser-chrome substitution class.** A harness affordance that substitutes the **user's physical input stack or browser chrome** (synthesized pointer-lock deltas or button events on a locked element, a forced native-Esc, a platform-API refusal) is **legitimate with declaration**, mirroring "the external may be the substitute": the device is an external; the product's **own handling of the input stays real** from the contracted interface onward. Each affordance is declared in the test-affordance register, never silently widened.
  - **An empty substitution set is stated, not omitted.** A product with no externals records that fact in the map; the substitution half of the staged DoD is then trivially satisfied, not vacuous.
- **Blocked-by-environment is a first-class gate-item state.** When a gate's binding environment is **structurally unavailable** to the build host (no display, no hardware GPU, no branded browser), the affected items take **both** recorded devices at once: they stay on the **open deferred-and-tracked list** *and* a **recorded accountable waiver** covers them (cause = the host cannot provide the named `ENV-###` fidelity · scope = the named gate items · re-run condition = execute on a conforming host). **Never a pass, never a silent skip** — deferral alone implies a scheduled future run; the waiver records that none is scheduled and who accepted that.
- **Off-gate evidence runs.** Where a gate is blocked, the same suites may run **off-gate** at the highest fidelity the host *can* provide — a named, **non-binding** run class: binding assertions off, the **observed fidelity recorded on the artifacts**, and the contracted gate configuration untouched. Evidence runs accumulate honest signal for a blocked gate; they never turn it green.

### Part 10b — Claude Code Operationalization (advisory only)

Markdown-first; tooling sits on top and never blocks.

- **Intake → scaffold skill** — runs the interview, writes the manifest, generates the doc set from templates with markers seeded.
- **Maturity-audit agent** (read-only) — reports each concern's rung vs section completeness; flags claimed-rung mismatches; lists gaps.
- **Level-up skill** — targets a concern, reads next-rung criteria, co-authors the delta, clears resolved markers.
- **Per-concern specialist agents** — deep on heavy concerns (security, UX, data, …).

### Part 10c — Relationship to the Delivery Methodology

The doc system hands off at **Contract-grade**; Delivery Process (11.6) owns building and the **Verified** rung. The delivery methodology in `docs/00-delivery-methodology.md` + `docs/00a-work-item-hierarchy.md` is the reusable core of the Delivery Process concern.

**The doc↔tracker boundary.** Where the delivery methodology's work items live in an external tracker (GitHub Issues, Jira, Azure DevOps, …), the tracker is a **downstream mirror of what Dictum owns**, not a second source of truth. Contracts and the `Verified`/build-status record stay **repo-authoritative**; work items **reference** contract IDs (owned-once, Part 5) and the repo stores no back-reference (a stored external edge would drift — Part 0.5; the only join is the stable ID). The tracker may *additionally* hold two things the repo doesn't own — **demand** (business intent, ingested via `doc-feature` — the request seeds the interview, Part 0.6, never substitutes for it) and **triage** (a suspected defect: an externally-sourced, unadjudicated `drift`-type change event (10d) that stays **repo-invisible until a reproducing case adjudicates it**). Two gates preserve integrity: nothing is **built** before it references a Contract-grade ID; nothing is **written to the repo** for a bug before reproduction (a code-defect verdict then revokes `Verified` via a regression case, leaving the contract text untouched — doc maturity ≠ implementation status). This boundary — the **execution mirror** — is implemented by a thin, product-local artifact, the **tracker-binding declaration** (channel, three item roles, two gates), owned by Delivery Process (11.6). Live two-way sync tooling is out of scope (`[FUTURE-SCOPE]`, ROADMAP).

### Part 10d — The Documentation Enhancement Lifecycle

A doc set is authored ~once (Part 0.1) and **enhanced** for the rest of the product's life — deepened, broadened, corrected, retired. It is a first-class topic, sub-numbered here because it extends the playbook past **Publish** and amends the **release gate** (10a). The key move adds *no new structure*: the **owned-once ID web** (Part 5), read backwards, **is** the change-impact graph — *who must change when a contract changes*. The lifecycle adds only a **change event**, a reverse traversal, and two pieces of durable state.

- **The change event (the unit of enhancement)** — raised against an affected stable ID; **source-agnostic** (a doc edit and a detected code↔doc drift emit the same event). It carries a **classification**:
  - `cosmetic` — wording or human **label** only (a label is mutable; the ID is not) → no propagation.
  - `additive` — new optional field / enum value / capability → **soft flag**: an informational `[REVISIT]` at referencers; **never blocks a gate** (each consumer decides whether it was breaking *for them*).
  - `breaking` — type / invariant / semantic change → **hard flag**: dependents marked **stale**.
  - `retiring` — contract removed → breaking + a **tombstone**, optionally `superseded-by: [ID…]` for split/merge. **IDs are never reused or renamed.**
  - **Shape (the contract both tools share):** `{ id, source: doc-edit|drift, classification, ref }` — where **`id` is the affected contract ID** and **`ref` is the changeset** (commit/PR). *(A defect filed in an external tracker enters here as a `drift`-type event — the tracker is a third **origin** of such events, alongside a doc edit and the drift-detector, so `source` stays `drift` — with `classification`/`direction` **proposed until a reproducing case adjudicates it**; see the doc↔tracker boundary, 10c.)* The event is keyed by `(id, ref)`; the detector also appends a **per-run** sequential ordinal (`<ref>#<n>`, numbered across all findings in the run regardless of contract) as a stable finding handle. A `doc-edit` event's `classification` is authoritative (author intent). A `drift` event adds `direction` (`code-ahead` = doc stale · `doc-ahead` = code defect · `divergent`), `evidence`, optional `facets` (dedup'd index findings such as a `binding-stale`), and `adjudication` — and its `classification` is *proposed*, not authoritative, until adjudicated. A pure-semantic `failing-inv` finding (no structural delta) classifies as `breaking` unless the invariant is advisory.
- **Propagation** —
  - **Derive** the reverse-reference set of the changed ID; never store the edge list — a second copy would itself drift, the exact failure Part 0.5 exists to prevent. Reference **granularity** for propagation (entity- vs field-level) is declared in the manifest (`reference_granularity`); it governs *how reverse-references are flagged only*, and is **independent** of whether the binding map carries field-level locators (those add drift-detection precision without changing propagation).
  - **One hop per change event.** A dependent propagates further only if re-examining it produces a *new* change event; confirmed-correct stops the cascade. Cycles reach a fixpoint — a non-terminating cycle signals two contracts that should be merged.
  - **Fidelity is rung-gated.** Propagation is reliable only between **Contract-grade+** nodes — that is where semantics live as checkable `INV-###` and a semantic break gains a *structural* footprint. Depending on a sub-Contract-grade contract is itself a `[GAP]`.
- **Durable state (hybrid).** The manifest holds only what git can't cheaply answer:
  - **Tombstones** (with optional supersession) — make a dangling reference a *detectable error* and enforce the **retirement precondition**: a tombstone can't finalize while live inbound references remain (unless they retire in the same changeset).
  - **Staleness** — a **cause-attributed set** of the change-event IDs that made a contract stale, *not* a boolean; it clears only when the set empties, so two concurrent changes can't clear each other.
  - History — classification, what changed, when — rides on the git changeset. No parallel copies (Part 6).
- **Gate interaction (amends 10a).** Staleness **does not lower the rung** — the verification record stands — **but blocks the release gate**, exactly like an open `[GAP]`. A **recorded waiver** can downgrade a stale flag to an accountable marker for a hotfix (the escape hatch that keeps the gate trusted). Over-flagging is *safe* (a re-verify) where under-flagging *ships a real break*; it is economically sustainable only when **Verified-rung checks are automated** (re-verify = re-run) — so the lifecycle is what turns automated verification into a compounding return.
- **Tooling (advisory, the 10b pattern).** The built `doc-change-impact` skill consumes the change event above: derive the impact set, write cause-attributed staleness, open markers, and **challenge a classification that understates a structural break**.
- **Drift-detector** — a **read-only advisory** agent (`agents/drift-detector.md`, model A) that compares code against the doc set and emits `source: drift` change events into the same interface; **it detects only — never edits or adjudicates.** *Binding-gated* (needs the contract→code **binding map**, below) and *rung-gated* (Contract-grade+ only). Detectable surface: *structural* divergence (signatures, fields, enum sets, routes, payloads) directly; *semantic* only via executable checks — a failing `INV-###` assertion **is** a drift signal. Suppresses contracts already flagged stale, scoped-out/`[FUTURE-SCOPE]`, or sub-Contract-grade, and honors the substitution rule (own real code only). **An in-repo divergence register does not suppress:** adjudication records living outside the manifest's staleness state (a findings register, an adjudication log) are not a suppression source — a finding matching an open register record is still **reported, once, with a dedup cross-reference** to that record, never re-opened as new drift. And a contract whose only verification lies inside a **recorded waiver's** scope is **undetectable under waiver** — a named detector state distinct from a coverage gap: reported as *unverifiable-here*, never silently passed and never counted conformant. An operator (or rule) **adjudicates** each event — `doc-stale` → realign the doc and propagate · `code-defect` → no doc change, a defect to the code tracker · `divergent` → split (prefer the *most specific* direction; reserve `divergent` for a genuine both-sides conflict). A **dangling binding** is reported as a distinct `binding-stale` finding — a defect in the *index*, not a code↔doc drift; it is fixed (or retired) in the binding map, with no doc/code adjudication. **One root cause → one event:** a single defect visible on several contracts is raised once (on the contract where it is most precisely observed — executable evidence such as a failing check is decisive over a merely-stated contract), cross-referencing the others, with dedup'd index findings carried as `facets`. Deterministic, CI-gradeable detection (model B) remains `[FUTURE-SCOPE]`.
- **Binding map** — the contract→code index the detector needs, a machine-readable sibling of the manifest (format + rules: `templates/binding-map.template.md`). It is a *stored* index, which Part 0.5 otherwise forbids; permitted here because it records **underivable** information (doc↔code correspondence cannot be derived from the doc set, unlike the reverse-reference graph) and is **self-validating**: every run checks the **doc end** (ID in-scope, Contract-grade+, not tombstoned), the **code end** (every locator resolves — a dangling one *is* a drift finding), and **coverage** (every code-realizable in-scope Contract-grade+ contract is bound; gaps are flagged, not detected).
- **Out of scope:** cross-product/platform **federation** of the reverse index.

### Part 10e — The Doc-Led Implementation Flow (the forward direction)

Part 10d runs two directions — **doc→doc** propagation (`doc-change-impact`) and **code→doc** drift (`drift-detector`). Part 10e adds the third, the one a *doc-as-source-of-truth* workflow lives on: **doc→code**, where an author **deliberately edits a contract to specify a new or changed feature, and the implementation must catch up.** It is the per-feature analogue of the first-build **build-ready gate** (Part 10): instead of "all in-scope concerns are Contract-grade → build the product," it is "this *delta* is Contract-grade → build the delta." It adds **no new axis and no stored state** — it reuses the change event (10d) and the binding map, and derives the rest.

- **The forward primitive is the `doc-edit` change event** (10d) — its classification is **authoritative** (author intent). A feature is a set of `additive`/`breaking`/`retiring` doc-edit events against new or changed contract IDs. (A pure-`cosmetic` edit creates no implementation work.)
- **Three input modes.** The flow's change set comes from **(a)** change events (a feature delta), **(b)** a doc-set **diff between two refs**, or **(c) first build** — an **empty binding map plus a passed build-ready gate (Part 10)**, where the change set is *every in-scope, Contract-grade, code-realizable contract* and every classification is trivially additive (`build-new` across the board). The first build is the degenerate whole-product delta; it needs no special machinery, only the recognition that "no bindings yet + gate passed" *is* a valid input.
- **The implementation backlog is *derived*, never stored** (the Part 0.5 discipline, exactly as for the reverse-reference graph). For each changed/added **code-realizable, Contract-grade+** contract, the binding map classifies the work:
  - **build-new** — the contract has **no binding** (a *coverage gap*) → new code + a new binding. *This is the same signal the drift-detector flags as "can't detect drift here"; the forward flow consumes it as "build this."*
  - **change-existing** — the contract is **bound** and hit by a `breaking`/`additive` edit → modify the code at the bound locators.
  - **retire** — a `retiring` event with a **tombstone** → remove the code and its binding, under the retirement precondition (10d).
  - **Suppressed** (not yet buildable): out-of-scope / `[FUTURE-SCOPE]` / **sub-Contract-grade** contracts. Depending on a sub-Contract-grade contract is itself a `[GAP]` to report, not to plan — the flow is **rung-gated**, like propagation.
- **Handoff to Delivery Process (11.6).** The backlog becomes **vertical slices** — Delivery's unit. Each slice realizes the delta's contract IDs through *every layer they touch* (the slice rule — never one layer), references the code sites (existing bound locators, or "new"), and carries the Quality coverage the delta owes (an assertion per new `INV-###`, a contract test per new `API-###`, real-flow `E2E-STANDARD` for a user-facing new `CAP-###`) plus the fidelity-staged DoD. Slices are **dependency-ordered** per the build playbook (Domain/Architecture → Interfaces → UX/downstream), so an owned contract exists before a slice references it. A contract may legitimately **span** slices (realized early, proven later); what is exact is that each contract has **exactly one *completing* slice** — the slice where its proof lands, recorded in Delivery's *Completes* column — never zero, never two.
- **The loop closes (Verified)** when the new/changed **bindings resolve** (the code exists), the **coverage-map tests pass**, and any **staleness** the change raised **clears** — at which point build-status (11.6) records the slice Verified and the drift-detector goes quiet (no coverage gap, no failing `INV`). Updating the binding map is part of the slice (Delivery territory), closing the same self-validating loop 10d relies on.
- **Symmetry — the two code↔doc tools share one interface.** `drift-detector` (10d) reads **code→doc** and reports *unintended* divergence; `implementation-planner` (10e) reads **doc→code** and plans *intended* catch-up. Both consume the change event + binding map; a **coverage gap is the shared signal** (flagged one way, planned the other). Neither stores state; both are **advisory** (they inform; Delivery builds).
- **Tooling (advisory, the 10b pattern).** `doc-feature` **skill** — author a feature delta into the owning concern(s) to **Contract-grade on the delta**, apply the document contract to each new/changed contract, classify it (emit the `doc-edit` event), wire the ID web, and **stub the binding map** (the unbound new contract is the planner's build-new signal). `implementation-planner` **agent** (read-only) — derive the backlog from the change events (or a doc-set diff between two refs, or the first-build mode above) + the binding map, and emit a **dependency-ordered slice plan** with code sites, required tests, and DoD. Where Delivery already **owns a normative build playbook** (a Contract-grade slice plan in the doc set), the planner still derives its plan **independently, then reconciles and flags divergences** — the owned playbook is the contract; the plan is advisory; silently adopting *or* overriding it would each break owned-once. **Out of scope:** generating the code itself — the planner *plans*; building stays Delivery's (human or AI).

### Part 10f — The Reverse-Authoring Flow (code→doc bootstrap, the brownfield entry)

The playbook (Part 10) enters **greenfield**: intake → scaffold → author-to-rung → build. But a doc set is often adopted onto a product that **already exists as code with no (or only informal) docs** — retro-adoption. Here the forward playbook fights the task: `doc-scaffold` is deliberately **code-blind** (it must never assume the product from the repo — correct for greenfield), and every existing contract is already **code-ahead** (the code exists, no doc does), so the two code↔doc tools can't help — `drift-detector` (10d) and `implementation-planner` (10e) are both **binding-gated and rung-gated over an existing doc set**. Part 10f is the missing third code-touching direction and the **bootstrap those two presuppose**: **code→doc at authoring time** — reverse-map the code into a faithful baseline doc set, so the product re-enters the ordinary lifecycle. It adds **no new axis and no new stored state** — it produces the ordinary manifest + doc set + binding map, just sourced from code plus a gap-filling interview.

- **The recoverability partition (what code can and cannot source).** The reverse flow's structure is that concern content splits three ways by code-recoverability:
  - **code-derivable** — Architecture, Interfaces & Contracts, Domain & Data, Quality mechanics, Delivery, Integrations, Operations, Observability: *structure lives in the code* (fields, signatures, routes, constraints, dep manifests, CI/IaC) and is **extracted**.
  - **partially derivable** — Product & Requirements (capabilities yes, **intent/personas/success** no), Security (mechanisms yes, **threat model/policy** no), User Experience (screens yes, **journeys/states/design intent** no), Performance (hints yes, **targets/SLAs** no): the structure is extracted, the intent is interviewed.
  - **not derivable** — Governance & Compliance, Business & Legal, and — categorically — **Non-goals & scope-out kinds** (Part 9): *code cannot distinguish "deliberately excluded" from "not built yet,"* so the calibration that Part 9 turns on is **un-extractable in principle** and must be interviewed.
- **Interview-first, strengthened — the code is a draft to react to.** This is not a weakening of Part 0.6; it is its best case. Extraction produces a **candidate** doc set the operator **confirms or corrects**, which is a higher-yield interview than blank-page generation. The confirm-and-fill loop still runs in rounds **to requirement saturation** — the prohibited "inferred enough" stop applies unchanged. Two tracks run: a **confirm track** (replay each extracted inference for confirmation) and a **fill track** (the partially-/non-derivable gaps above).
- **Assumption discipline (the reverse flow's honesty).** Every code-inferred fact is provisional until confirmed → recorded as an `[ASSUMPTION]` (Part 6). Confirmation clears it; an **uncorrected** inference **stays an `[ASSUMPTION]` in the published doc**, visible for audit. This is precisely the *inference-then-confirm* form of the autonomous-mode concession (Part 0.6) — except an operator **is** present, so inference is a starting draft, never the settled answer. An unrecoverable-and-unanswered fact is a `[GAP]`, never a guess.
- **Document as-built, but never enshrine smells.** Reverse-authoring adopts the code's *actual* contracts — but it is **not blind transcription**. Suspected defects, code↔informal-doc contradictions, dead code, and undocumented edge behavior are recorded as `[REVISIT]`/Open Questions, **never minted silently as contracts and never silently "fixed."** Enshrining a bug as a contract, or quietly correcting one, would each corrupt the baseline.
- **Scaffolded-but-unbuilt ("aspirational") contracts are the dominant brownfield signal.** On a real repo the largest gap is between *documented/scaffolded intent* and *implemented behavior*: an orphan schema table nothing reads, a README-claimed feature with no code, a UI call to a route the backend never defines. Such a contract has **structure without behavior** — it is **neither delivered nor a mere smell**, but the concrete contract-level instance of Part 9's *deliberately-excluded-vs-not-built-yet* ambiguity, which code cannot resolve. The reverse flow **must not mint it as a delivered/Verified contract** (its structure existing ≠ its behavior existing); it is surfaced for the interview and resolved to **`deferred`** (a `[FUTURE-SCOPE]` re-entry note owed), **`absent`** (dropped, the dead structure a `[REVISIT]` for the code owner), or a **defect** (the behavior is *owed* — a `[GAP]`). It carries **no behavioral binding** — it is a coverage gap by construction, which is exactly the "build this" signal the forward tools (10d/10e) then consume. *(Dominant on **messy** repos — but "none found" is a first-class valid result: a mature, disciplined codebase has no orphans, and the reachability check is as much a guard against **false** aspirational flags as a finder of real ones.)*
- **Scope boundary — non-software deliverables.** Reverse-authoring maps *software* contracts. A repo that also carries **firmware / embedded / hardware / mechanical** deliverables (a microcontroller sketch, a HID report format, a pin↔logical-input map — which can be a genuine cross-boundary invariant — a bill of materials, mechanical models) exposes surface Dictum's trait/ID vocabulary does not yet model. The flow **scopes these out honestly** (named in Non-goals as `absent`/`deferred` with the trait fact as justification, Part 9) rather than silently dropping them or forcing them into a software kind. Extending the vocabulary to a first-class embedded/hardware surface is a parked design item (ROADMAP).
- **The binding map is a first-class byproduct, and it resolves the code-ahead baseline.** Greenfield mints `bindings.yaml` slice-by-slice *during* the build (10d); brownfield mints it **at authoring time**, because the code already exists — the single largest payoff of the reverse flow, since it immediately unlocks `drift-detector` and `implementation-planner` on a repo that had neither doc nor map. Adopting code-as-built into the docs turns the initial "everything is code-ahead" condition into the **fixpoint** drift-detection needs. Existing passing tests are recorded as **candidate `Verified` evidence for Delivery to adjudicate** — the reverse flow never *claims* the Verified rung (it is Delivery's, Part 3; a green test over an unconfirmed contract proves nothing about intent). Run against such a fresh baseline the `drift-detector` (10d) degenerates, by design, into a **binding-map validator + invariant test-runner** — the structural comparison set is near-empty because every concern is still sub-Contract-grade, so it rolls the blanket premature-binding warnings up to one informational line and its value is the bound-`INV` pass/fail evidence; this is the expected bootstrap mode, not a tool failure.
- **Annotations buy identity, not knowledge — a recommended build-time practice.** Reverse-authoring recovers *what exists* (entities, endpoints, invariants, components, dependencies) at high fidelity whether or not the code carries any Dictum markers — **structure is annotation-independent**. What a code artifact **cannot** yield on its own is each contract's **stable ID**: an un-annotated pass must *coin* fresh IDs from code names, so the recovered set's identities diverge from any prior authoring and the owned-once web has no shared key to join on. The cheap fix is a **build-time convention that tags each realized contract's ID at its code site** (a `// DICT: <ID>` comment on the handler/struct/test, or the ID in the test title). It costs almost nothing while building and converts a later code→doc pass from a *fresh mint* into a *recovery* of the author's own IDs — which also sharpens `drift-detector` and lets the binding map be re-derived. *(Validation — reverse-authoring the same codebase with vs. without such annotations: structure recovered ~equally either way, but exact-ID recall fell from roughly half to near-zero without them, invariant/decision identity was lost, and re-adoption cost went from mechanizable to manual.)* The in-code tag complements the binding map (which stores the same ID→code link out-of-band); it is the always-in-sync, human-visible form.
- **Re-adoption owes an ID-reconciliation step.** A from-scratch excavation coins a **new candidate ID space** and tends to **over-fragment** (one authored contract surfacing as several finer code-level ones), so a regenerated set is **never a drop-in replacement** for an existing doc web. Adopting one onto an incumbent web requires: **join** regenerated IDs to incumbent IDs (trivial where in-code annotations survived, by body-comparison where not); emit a **tombstone-alias table** (Part 10d — never silently reuse or drop an ID); **roll up** over-fragmented contracts to the incumbent granularity; and **flag every unmatched contract** for human adjudication. Absent an annotation trail this step is manual and dominates re-adoption cost; budget for it explicitly. (First-bootstrap onto an *empty* web needs none of this — there is nothing to reconcile.)
- **Structure is a hierarchy, and granularity is *controlled*, not rolled up.** Code sits at a *finer* grain than a hand-authored contract set (many endpoints where an author wrote one capability). The author's coarser grain encodes **intent** — which consumer, which grouping — that code cannot recover, so collapsing the extraction to match it is both lossy and a guess (and for an AI implementer the finer, more-explicit grain is often the *better* one). The reverse flow therefore **keeps contracts at the fine code grain by default** (maximal explicit context; each contract independently drift-checkable) **and synthesizes a coarse index over them** — `CAP`/resource groupings wired to their constituents through the **reference web** (`CAP` realized-through → its `API`/`SCREEN`; `SCREEN → API`; `API → ENTITY`; raise/consume → `INV`/`EVT`), reconstructed from the code's own call graph / imports / data flow rather than left as bare nodes. Granularity is then a **view over one linked structure**, not a destructive choice: a manifest `authoring_granularity` (default `fine`) selects the *primary* authored level, and because contracts are **owned-once at each level and cross-linked**, drift-detection and re-adoption reconciliation operate at whichever grain. The failure to avoid is **not** "too many contracts" — it is contracts emitted **flat and unlinked**: nodes without the web are un-navigable for a human and un-traceable for the method, at any count. Reconstructing the web is the same work whether the grain is fine or coarse, and it is where most of a map's value lives.
- **Honest rung outcome — a baseline, not build-ready.** Extraction lands most concerns at **Sketch/Specified** with a structural Contract-grade surface; the fill-track interview raises them. The reverse flow's product is a faithful **baseline**, and it **hands off** (to `doc-levelup`/`concern-specialist`) to reach the build-ready gate — it does not assert build-readiness itself. Claiming Contract-grade off extraction alone would inflate the rung past what the intent-level sections actually contain.
- **Tooling (advisory, the 10b pattern).** `code-cartographer` **agent** (read-only) — maps the repo, derives traits, extracts candidate contracts by kind (each with evidence, code locators, confidence), drafts the binding map, assesses existing tests/CI, and emits the **recoverability report** (per concern: high/partial/none + the must-interview gaps) and a **smell list**; it **extracts and reports**, never authors or adjudicates. `doc-excavate` **skill** — runs the cartographer, then the confirm-and-fill interview, and **writes** the manifest (with reverse-authored **provenance**), the doc set (as-built, `[ASSUMPTION]`/`[REVISIT]`/`[GAP]` seeded), the `bindings.yaml`, and the derived index. **Out of scope:** making the set build-ready (that is the fill-track + `doc-levelup`) and editing the product code (the flow reads code, writes only docs).
