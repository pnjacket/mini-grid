# Concern 11.10 — Operations & Infrastructure

> **One-line:** How the system is physically deployed, configured, provisioned, run, and recovered — and the environment fidelity map that says what's real vs substituted at each stage.

## Purpose & Scope

Active when the product is deployed as a running service — **or, more generally, for any product whose staged DoD needs an environment fidelity map** (Part 10a requires one). A client-only product with no server still has owned environment truth: the browsers/runtimes it must run on, the fidelity at which tests run against them, and the serving contract for its static artifact — so the trigger is "an `ENV-###` map is needed", not "a service runs". Owns the **physical realization** of the `COMPONENT-###` model and the **environment fidelity map** (round 22). It does **not** own telemetry (Observability & Monitoring).

## Owned Contracts (referenced by stable ID)

1. **Environment fidelity map** — `ENV-###`: each environment declares, per external dependency, real vs approved-substitute, and an explicit **`auth:`** assertion as a checkable per-environment field, not prose — `real, no bypass` where an auth surface exists, or the enumerated value **`n/a — no auth surface exists`** for a no-auth product (the field stays checkable either way; it is never simply absent). Fidelity **may differ by test tier within one environment** (e.g. a real S3 container at the gate's E2E but a Mock-S3 in unit tests) — when it does, the **gate's E2E fidelity is what binds**. Alongside the per-external column, the map names the **platform/host fidelity dimension**: the *runtime itself* can be the substituted thing (a software-GL renderer standing in for hardware GPU, a headless engine for a headed display) — and the **browser/runtime build is itself a fidelity fact** to record (a harness-bundled or patched engine is a *proxy* for the branded browser the contract names). **Referenced by** Quality & Testing (maps test stages) and Delivery Process (the staged DoD's merge/release gates). The single source of environment truth.
2. **Deployment spec per `COMPONENT-###`** — build, deploy unit, scaling, resources, runtime config. Architecture owns logical; Ops owns physical; `COMPONENT-###` is the join.
3. **Runbooks** — `RUNBOOK-###`: deploy, rollback, recovery, incident.
4. **Dev-toolchain register** — `TOOL-###`: the version-pinned **build/test-time-only tooling** (compiler/type-checker, test runner, static dev server, automation harness). Such tooling is **not** a `DEP-###` (11.12 owns *runtime* externals); its register lives **here**, co-owned with Quality & Testing for the tiers that run it — including (especially) when 11.12 is out of scope entirely.

## Consumed (referenced, not owned)

- `COMPONENT-###` (Architecture) — the logical units being realized.
- Integrations & External Dependencies (11.12) — the externals the fidelity map substitutes.

## Contract-grade bar

Environments + fidelity map + per-`COMPONENT` deployment spec + runtime config + reproducible provisioning + key runbooks (deploy/rollback/recovery).

- **Fidelity may differ by test tier within one environment.** The `ENV-###` map records the dependency's fidelity at each tier; where a dependency is real at the gate's E2E but mocked in unit tests, **the gate's E2E fidelity is what binds** the staged DoD.
- **`auth:` is a checkable field with an enumerated no-auth value.** Each `ENV-###` carries the assertion explicitly per environment rather than relying on prose — `real, no bypass` where an auth surface exists, `n/a — no auth surface exists` where none does — so a reviewer can confirm no environment ships an auth shortcut *and* a no-auth product doesn't leave the field undefined.
- **The platform/host is a fidelity dimension.** The map records not only per-external fidelity but the fidelity of the runtime itself (software vs hardware GL, headless vs headed, harness-bundled engine vs branded browser build) — the runtime can be the substituted thing, and the browser/runtime *build* is a fidelity fact in its own right.
- **Multi-store backup unit = the consistent set.** When durable state spans multiple stores, the backup/restore unit is the **consistent set** (e.g. the database dump paired with the object-store snapshot), not any single store; a backup of one store alone is not Contract-grade.
- **Real-time/WS products state an ingress/edge concern.** Protocol upgrade and ingress routing are explicit; **session affinity is not required when fan-out is external** (a shared pub/sub broker delivers across nodes), and the map should say so rather than leave affinity implied.

## Worked Example (a planning SaaS)

`08-deployment.md` is this concern. At *Contract-grade*:
- **Environments + fidelity map:** `ENV-DEV` (local k8s + LocalStack — S3/KMS/SQS = approved substitutes), `ENV-QA` (same), `ENV-STAGING`/`ENV-PROD` (real AWS — S3/KMS/SQS = real). Each carries **`auth: real, no bypass`**. S3 is real at the gate's E2E tier but a mock in unit tests — the E2E fidelity is what binds. This is exactly the map the merge gate (dev) and release gate (prod) reference.
- **Per-`COMPONENT` deployment:** each Go service → distroless image + Helm sub-chart; each Angular app → nginx image; resource/scaling per component.
- **Backups (multi-store):** the restore unit is the **consistent set** — the PostgreSQL dump paired with the object-store bucket snapshot (attachments reference `ENTITY-ATTACHMENT.storage_key`), not the database alone.
- **Ingress/edge (real-time):** ingress terminates TLS and handles the WebSocket protocol upgrade; **no session affinity** is configured because fan-out goes through an external pub/sub broker, so any node can serve any socket.
- **Runbooks:** `RUNBOOK-KEYCLOAK-BOOTSTRAP`, deploy via `make up`, rollback.
- *Note:* the local k8s cluster pre-exists; there's no AWS account, so `ENV-STAGING`/`PROD` are deferred to the AWS epic (`[FUTURE-SCOPE]`).

For a **client-only** product (e.g. a client-only browser game) the same vocabulary degenerates cleanly in one line: **deploy unit = the static docroot** (the built site directory, servable by any static host); per-component build = the bundling step; scaling/resources = n/a — stated, not omitted; the `ENV-###` rows carry browser/runtime fidelity (engine build, GL backend) and `auth: n/a — no auth surface exists`.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Ops owns the `ENV-###` environment fidelity map. | Closes the round-22 loop: the staged DoD and test-stage mapping need one authoritative declaration of what's real vs substituted, and it belongs with the concern that provisions environments. |
| 2 | Ops = physical realization; Observability = telemetry. | Keeps deployment/runtime separate from logging/metrics; both keyed off the running-service trigger. |
| 3 | Deployment specs are keyed to `COMPONENT-###`. | Logical↔physical traceability: every architectural component has a known deploy unit, and no deploy unit is unaccounted-for. |
| 4 | Contract-grade requires reproducible provisioning + recovery runbooks. | "It runs on my cluster" isn't operations; reproducibility and a tested recovery path are what make a service operable by someone other than its author. |
| 5 | Fidelity may differ by test tier within one environment; the gate's E2E fidelity binds. | "Real at prod / faked at dev" is too coarse — a dependency can be real in the gate's E2E yet mocked in unit tests. Recording fidelity per tier, and binding on the E2E tier, keeps the staged DoD honest. |
| 6 | `ENV-###` carries an explicit `auth:` assertion, with `n/a — no auth surface exists` as an enumerated value. | "No auth bypass" stated as prose can't be checked; a per-environment field can. The no-auth value exists because the field as originally defined presumed an auth surface — a no-auth product had no conformant way to fill it, and an absent field is indistinguishable from a forgotten one. |
| 7 | Multi-store backup unit is the consistent set. | A correct restore must pair stores (DB dump + object-store snapshot); backing up one store alone yields dangling references and silent data loss. |
| 8 | Real-time/WS products get a `network-ingress-edge` sub-aspect. | Protocol upgrade and ingress routing have no home in the deployment-spec vocabulary; naming the sub-aspect also lets the map record that session affinity isn't required when fan-out is external. |
| 9 | The trigger is broadened: any product whose staged DoD needs an `ENV-###` fidelity map activates this concern. | A client-only browser product is no "deployed running service", yet Part 10a still requires an owned fidelity map and someone must own the browser-support/serving contract — a validation build had to pull Ops in by operator override. The trigger now names the actual need. |
| 10 | The fidelity map has a **platform/host dimension**; the browser/runtime build is itself a fidelity fact. | With zero externals, the map's per-external vocabulary was empty while the *binding* fidelity question (software vs hardware GL, harness-bundled engine vs branded browser) had no named home. The runtime can be the substituted thing — recording it is what keeps a "passes on ENV-002" claim honest. |
| 11 | Build/test-time-only tooling registers here as `TOOL-###`, co-owned with Quality — never as `DEP-###`. | Dev/test tooling (test runner, compiler, static server) is not a runtime external, so 11.12 rightly excludes it; but when 11.12 is out of scope entirely the pins had no prescribed home and a validation build minted `TOOL-###` locally. Ops owns environments, and the toolchain is part of what an environment *is*. |
