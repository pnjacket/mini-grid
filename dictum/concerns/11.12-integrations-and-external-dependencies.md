# Concern 11.12 — Integrations & External Dependencies

> **One-line:** Everything the product depends on but doesn't own — how we call it, what we assume, how it fails, what we fall back to, and how it's substituted in lower environments.

## Purpose & Scope

Active when the product depends on third-party services/APIs/SDKs **at runtime**. Owns the **outbound** (consumed) contracts; Interfaces owns the **inbound** (exposed) ones. Per-dependency failure handling lives here; the reusable resilience mechanism lives in Architecture. **Build/test-time-only tooling (a test runner, a compiler, a static dev server) is *not* a `DEP-###`** — it is part of what an environment is, so its version-pinned register is owned by Operations (`TOOL-###`, 11.10), co-owned with Quality & Testing; scoping this concern out therefore scopes out *runtime* externals only, never the toolchain pins.

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Default (when deps exist) | Notes |
|---|---|---|---|
| Per-external contract (`DEP-###`) | `per-external-contract` | in | How called, auth, assumptions |
| Failure modes + fallback/degradation | `failure-modes-fallback-degradation` | in | Timeout/retry/circuit-break + degraded mode |
| Criticality | `criticality` | in | App-fatal vs capability-degrading vs fail-soft (per dependency) |
| Fidelity substitution | `fidelity-substitution` | in when substitutable | Links Ops `ENV-###` fidelity map |
| Version pinning | `version-pinning` | in | Pinned versions/compat |
| Data mapping (external ↔ `ENTITY-###`) | `data-mapping` | optional | |

## Owned Contracts (referenced by stable ID)

1. **External dependency** — `DEP-###`: contract + auth + failure modes + fallback + **criticality** + approved fidelity substitution + pinned version. **Referenced by** Operations (the `ENV-###` fidelity map substitutes these; the criticality field tells the liveness/readiness gate which deps are app-fatal) and the staged DoD.

## Consumed (referenced, not owned)

- `PATTERN-###` (Architecture) — resilience mechanisms applied per dependency.
- `ENV-###` fidelity map (Operations) — where substitutions are declared.
- `ENTITY-###` (Domain) — for data mapping.

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Integrations & External Dependencies |
|---|---|
| **Absent** | No external dependencies (concern not in scope). |
| **Sketch** | A list of externals, as bullets. |
| **Specified** | Each described with how it's called + auth; failure noted. |
| **Contract-grade** | `DEP-###` with contract + auth + failure modes + fallback + criticality + fidelity substitution + version pin. An implementer can call it, handle its failure, knows whether its failure is app-fatal, substitute it in lower environments, and knows the pinned version. |
| **Verified** | *(delivery methodology)* Integrations work against real + substitute; failure handling tested. |

## Contract-grade bar

Per external: contract + auth + failure modes + fallback + **criticality** + fidelity substitution + version pinning.

- **Criticality (load-bearing for a multi-dependency product).** Each `DEP-###` states its criticality dimension — **app-fatal** (its absence gates liveness/readiness; the service must not report ready without it), **capability-degrading** (one capability is unavailable but the rest of the product runs), or **fail-soft** (the call degrades silently). For a product with several externals this — not a generic "failure modes + fallback" footnote — is the decision an operator needs to wire the health probes and the on-call response; a flat list of dependencies without it is under-specified.
- **Substitution may be "the external *is* the substitute."** The fidelity substitution field blesses the case where a self-hosted stand-in *is itself* the approved dependency, run **real in every environment** (no prod-real / dev-fake axis); the eventual vendor swap is `[FUTURE-SCOPE]` behind the **same** `DEP-###` contract. The default "real at prod, faked at dev" framing does not fit these and must not be forced onto them — recording the stand-in as the real dependency everywhere, with the swap deferred behind the contract, is Contract-grade.

## Worked Example (a planning SaaS)

The external dependencies are AWS services, Keycloak, and PostgreSQL. At *Contract-grade*:
- `DEP-AWS-S3` — object storage; auth via IAM; failure → retry/backoff; **criticality: capability-degrading** (uploads/attachments unavailable, core app runs); **substituted by LocalStack at `ENV-DEV`/`ENV-QA`** (per Ops fidelity map), real at staging/prod; version pinned.
- `DEP-KEYCLOAK` — OIDC; failure → fail-closed auth; **criticality: app-fatal** (no auth → not ready); pinned to v26.
- `DEP-POSTGRES` — primary store; failure → connection retry; **criticality: app-fatal** (gates readiness); pinned major version.
- *External-is-the-substitute example:* `DEP-OBJECT-STORE` realized as **self-hosted MinIO run real in every environment** (S3-compatible); the managed-S3 swap is `[FUTURE-SCOPE]` behind the same `DEP-OBJECT-STORE` contract, so there is no prod-real/dev-fake axis to record — the stand-in *is* the dependency.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | `DEP-###` per external, with substitution + failure handling. | Makes each dependency a complete contract, and the substitution field is what the staged-DoD fidelity map keys off. |
| 2 | Integrations owns outbound; Interfaces owns inbound. | "What we assume of others" and "what we promise" are different contracts reasoned about differently; the direction split keeps each clean. |
| 3 | Per-dependency failure handling here; reusable resilience patterns in Architecture. | Specifics (this API's retry budget) belong with the dependency; the mechanism (circuit breaker) is cross-cutting. |
| 4 | Contract-grade requires fallback + fidelity substitution. | External failure and "can't run it locally" are the two ways integrations bite in practice; mandating both closes them. |
| 5 | `DEP-###` carries an explicit criticality dimension (app-fatal / capability-degrading / fail-soft). | For a multi-dependency product the load-bearing decision is *which* failures gate liveness/readiness vs degrade one capability vs fail soft. A generic "failure modes + fallback" footnote doesn't surface it; the criticality field is what the health probes and on-call response key off. |
| 6 | Substitution blesses "the external *is* the substitute." | A self-hosted stand-in run real in every environment (vendor swap deferred behind the same contract) has no prod-real/dev-fake axis; forcing the "real at prod, faked at dev" framing onto it misrepresents the dependency. |
| 7 | `DEP-###` is **runtime** externals only; build/test-time tooling registers as Ops-owned `TOOL-###`. | A zero-runtime-dependency product legitimately scoped this concern out, leaving its dev tooling (automation harness, type-checker, static server) with no prescribed pinning home — the standard was silent and the build minted a local prefix. Naming the boundary and the owner (11.10 + Quality) keeps "11.12 out of scope" from silently dropping the toolchain pins. |
