# Concern 11.14 — Accessibility & Internationalization

> **One-line:** Whether everyone can use it (a11y) and whether it works across languages and regions (i18n) — two independent sub-aspects that switch on separately.

## Purpose & Scope

**a11y** activates with any human-facing UI; **i18n** activates with multi-locale scope — either can be on alone. First-class concern that binds its requirements to UX screens and locale-affected entities. Tested via Quality's a11y/i18n test types.

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Activated by | Notes |
|---|---|---|---|
| a11y conformance (per `SCREEN-###`) | `a11y-conformance` | a11y (any UI) | WCAG level; keyboard, contrast, semantics, focus; **for a live-updating UI, the accessible-announcement contract** (below) |
| String externalization | `string-externalization` | i18n | No hard-coded user-facing text |

### Accessible-announcement contract (live-updating UIs)

The facet list above (keyboard / contrast / semantics / focus) is **static-DOM** a11y. A product with a **live-updating UI** — content that changes without a user action (a server-pushed message arriving, a status changing) — has an a11y obligation those facets never cover: a screen-reader user must learn the content updated **without losing their place**. This is the hardest a11y obligation for a real-time app, and the static-DOM bar passes while it is entirely absent.

For any live-updating UI, the a11y Contract-grade bar **additionally** requires an **accessible-announcement contract** for asynchronous content updates:

- **An ARIA live region** announces incoming updates, with a **stated politeness** (`polite` vs `assertive`) per kind of update.
- **Without stealing focus** — the announcement reaches assistive tech but the user's caret/focus stays where it was.
- **Burst throttling / coalescing** — under a burst (many messages at once) announcements are throttled/coalesced rather than read one-by-one.
- **Deliberate exclusions** — high-churn ephemeral events are **explicitly excluded** from announcement; the contract names what is silent and why. The messaging instances are typing and presence; **non-messaging churn is the same exclusion kind** — frame/telemetry readouts, periodic timer events, background content streaming (e.g. a game's world loading) — whatever updates constantly *by design* and would bury real announcements.
- **Ownership split with the node's owner** — the concern that owns a DOM node fixes the *node* (structure, roles, names); **whether content inserted into a live region is announcement-eligible belongs to *this* contract**, keyed to the update's trigger/provenance (a user-initiated save announces; a timer-driven one may be deliberately silent). The node owner must not pre-decide eligibility by where it places content.

This binds to the live-UI **event-driven/ambient states** UX owns (11.9): a `EVT-S-MESSAGE-NEW` that paints an ambient state must also be reachable to a screen reader by this contract.

## Owned Contracts (referenced by stable ID)

a11y contracts use the **`A11Y-*`** ID family. Each is **owned per `SCREEN-###`** (or per the persistent SPA-shell where a contract is global, e.g. the live-region announcement contract): a per-screen `A11Y-###` carries that screen's conformance specifics, and where it matters the announcement contract.

1. **a11y requirements per `SCREEN-###`** (`A11Y-###`) — conformance level + per-screen specifics + (live UI) the accessible-announcement contract.
2. **i18n requirements** — per `SCREEN-###` and per locale-affected `ENTITY-###`.

## Consumed (referenced, not owned)

- `SCREEN-###` (User Experience); locale-affected `ENTITY-###` (Domain); Quality's a11y/i18n test types.

## Contract-grade bar

Conformance target per screen (`A11Y-###`) + **(if the UI is live-updating) the accessible-announcement contract** (ARIA live region with stated politeness, no focus-steal, burst throttling/coalescing, and named exclusion of high-churn ephemeral events) + (if i18n on) externalized strings + locale formats + RTL + pluralization + translation process.

- **A conformance claim = level X minus traced SC exceptions.** Each excepted success criterion is named and traced to the scope-out that creates it — and that scope-out may legitimately live in *another* concern (e.g. UX scoping out `responsive-layout` punches a justified hole in a reflow SC). A **traced** cross-concern hole is legal; an untraced silent shortfall is what fails the bar. The claim's value is that its exceptions are enumerable, not that it is exceptionless.
- **A focus-placement rule states *when* it binds.** Where the contracted landing element depends on an **asynchronously-resolved enabling state** ("the primary button *if enabled*, else the fallback"), the contract says whether the landing is asserted **at state entry** (per the then-current DOM) or **after the enabling read settles** — and a late re-application may only move focus that still sits where arrival placement left it, never focus the user has since moved. Silent on this, the contracted element is unknowable at placement time and the test races the read.
- **A recommended mechanism is checked against sibling concerns' test surfaces.** A mechanism this concern *recommends* (e.g. `inert` for modal background containment) can collide with another concern's E2E-verified contract on the same DOM (e.g. a UX capture-at-interaction behavior exercised by editing a background field). The contract here is the **observable** ("Tab never reaches outside the open layer"); the mechanism stays delegated, and a recommendation that breaks a sibling's verified surface yields to any mechanism satisfying the observable.

## Worked Example (a planning SaaS)

Illustrates the independent sub-triggers:
- **a11y ON** (it has a UI): WCAG 2.2 AA bound per screen as `A11Y-GRID-VIEW` — `SCREEN-GRID-VIEW` must be keyboard-navigable with sufficient contrast and proper grid semantics. (this planning grid is **not** live-updating, so the announcement contract is out of scope here; a real-time product such as a chat app would carry an `A11Y-*` announcement contract on its message-list screen: `polite` live region for `EVT-S-MESSAGE-NEW`, no focus-steal, coalesced under a burst, silent for typing/presence.)
- **i18n likely OFF** for v1 (single locale): the sub-aspect is out-of-scope until, say, EU expansion turns it on — at which point string externalization, locale formats, and RTL become in-scope without touching a11y.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Two independent sub-triggers (a11y: any UI; i18n: multi-locale). | A single-locale tool still has a11y obligations; a CLI localized for many regions has i18n but limited a11y. Coupling them would mis-scope both. |
| 2 | a11y conformance bound per `SCREEN-###`. | Per-screen requirements are checkable; a product-wide statement isn't. |
| 3 | First-class, not folded into UX. | Keeps the round-6 decision; a11y/i18n have their own rubric and tests rather than being buried as UX footnotes. |
| 4 | Contract-grade scopes to active sub-triggers. | You pay for i18n only when you localize and a11y only when there's a UI — the scope lever applied within one concern. |
| 5 | A live-updating UI must carry an accessible-announcement contract. | Static-DOM a11y (keyboard/contrast/semantics/focus) is silent on content that updates without a user action; announcing live updates politely, without stealing focus, throttled, and silent for typing/presence is the hardest a11y obligation for a real-time app and the bar would pass with it entirely absent (failure-mode #17). |
| 6 | a11y contracts use the `A11Y-*` family, owned per `SCREEN-###` (or per SPA-shell when global). | a11y had no contract-ID scheme — unlike Interfaces/Domain which hand you a prefix — so the family was being invented locally; naming it closes the gap and lets a11y contracts be referenced on the ID web like any other. |
| 7 | Exclusion vocabulary covers **non-messaging churn** (timers, telemetry, content streaming), not just typing/presence. | The exclusion examples were messaging-shaped; a client-only game's constant churn (frame telemetry, world streaming, periodic timers) is the same kind — by-design noise that would bury real announcements — but read as uncovered, so the build had to argue the analogy (failure-mode #22). |
| 8 | A conformance claim is **level X minus traced SC exceptions**, and a scope-out in one concern may punch a justified hole in this concern's bar. | A WCAG-AA claim contains criteria whose preconditions another concern scoped out (responsive layout); without traced exceptions the claim is either falsely absolute or silently weakened. Tracing keeps cross-concern holes accountable instead of invisible. |
| 9 | The node owner fixes the node; **content-insertion eligibility belongs to the announcement contract**. | The owned-node vs owned-announcement split leaked: a node owner deciding where content lands inside a live region thereby pre-decided what announces. Assigning insertion-eligibility to this contract (keyed to update provenance) keeps the decision where it is reasoned about. |
| 10 | Focus-placement bars state their binding moment relative to async enabling reads; mechanism recommendations are checked against sibling concerns' test surfaces. | A title-arrival focus rule raced an async has-save read (the contracted element was unknowable at placement time), and a recommended `inert` containment collided with a sibling UX contract E2E-verified by editing a background field. Both are contract-precision gaps in this spec, not product bugs. |
