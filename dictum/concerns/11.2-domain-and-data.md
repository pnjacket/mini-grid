# Concern 11.2 — Domain & Data

> **One-line:** What the data *is*, what rules it must obey, and how it persists. Owns the canonical entity definitions the rest of the set is shaped against.

## Purpose & Scope

Defines the conceptual **domain model** (entities, relationships, invariants) as the canonical truth, and the **storage model** that realizes it. Owns the `ENTITY-###` definitions and field-level classification tags. It does **not** own protection/retention *policies* (Security & Governance) nor the wire shapes that project from entities (Interfaces).

## Owned Contracts (referenced by stable ID)

1. **Entity definitions** — `ENTITY-###`: fields, types, identifiers, relationships. **Referenced by** Interfaces (DTOs/wire shapes project from these), User Experience (what's displayed/edited), Security & Privacy + Governance (act on classification tags), Quality & Testing (test data shape). Where user accounts / identity exist in the product, includes the **user entity** `ENTITY-USER`, linked to `PERSONA-###` (Product & Requirements); a product with no accounts (e.g. a client-only browser game) has no `ENTITY-USER`, and its absence needs no Non-goals defense.
2. **Invariants** — `INV-###`: business rules stated as checkable conditions. **Referenced by** Quality & Testing (each becomes assertable).
3. **Classification tags** — field-level `pii` / `sensitive` / `secret` flags. **Referenced by** Security (protection policy) and Governance (retention/residency policy).

## Consumed (referenced, not owned)

- `PERSONA-###` (Product & Requirements) — `ENTITY-USER` links to personas.

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Domain & Data |
|---|---|
| **Absent** | No data model. |
| **Sketch** | Key entities and relationships as bullets. |
| **Specified** | Entities described with fields, relationships, and lifecycle in prose; invariants stated narratively; storage approach noted. |
| **Contract-grade** | `ENTITY-###` with typed fields + identifiers (**id format specified, incl. UUID version where it carries semantics**); relationships explicit; invariants as checkable `INV-###` (**a uniqueness / exactly-once `INV-###` names its enforcement mechanism**; an `INV-###` that depends on a derived key specifies the derivation); lifecycle states defined; sensitive fields tagged; storage schema specified (when persistence is in-scope). An implementer can build the model and a test can assert its invariants without guessing. |
| **Verified** | *(delivery methodology)* The built schema enforces the invariants; tests assert them. |

## Contract-grade bar

Entities + types + stable IDs + relationships + checkable `INV-###` invariants + lifecycle states + classification tags (and storage schema if persistence is in-scope). Three additional bars on identifiers and invariants:

- **Enforced ≠ checkable.** An `INV-###` that asserts **uniqueness or "exactly once"** must **name its enforcement mechanism** — a DB constraint, a unique index, a transaction, or a **structural / by-construction** mechanism in stores that have no declarative constraints (a fixed primary key, a single-record `put()`, a transaction scope): what matters is that the store *cannot represent* the violation, not which feature prevents it. *Checkable by a test* is not the same as *enforced by the schema*: an invariant with only a test behind it can still be violated under concurrency, so an unenforced uniqueness/exactly-once invariant must be explicitly marked advisory. (failure-mode #18a.)
- **An identifier's format is part of its contract.** State the id format precisely, **including the UUID *version* where the version carries semantics** (e.g. a time-ordered v7 id underpinning an ordering `INV-###`). The version is owned here; downstream validators (Interfaces 11.4) must honor it. (failure-mode #18b.)
- **A derived key is itself a contract.** When an `INV-###` depends on a **derived key** (e.g. a canonical key computed from a sorted member set), specify the derivation **once** — the inputs, ordering, encoding, and delimiter — so independent modules computing it converge.

## Worked Example (a planning SaaS)

`05-model-specification.md` is this concern at ~*Specified*. To reach *Contract-grade*:
- **Entities:** `ENTITY-TENANT`, `ENTITY-USER`, `ENTITY-MODEL`, `ENTITY-ATTRIBUTE`, `ENTITY-SCENARIO`, `ENTITY-CALCULATION` — each with typed fields + identifiers.
- **Invariants:** `INV-CALC-DAG` (calculation references form a DAG — no cycles), `INV-TENANT-ISOLATION` (every row resolves to exactly one tenant).
- **Lifecycle:** model `Not Configured → Configured`; soft-delete → hard-delete after grace period.
- **Classification:** `ENTITY-USER.email = pii`.
- **Storage realization:** schema-per-tenant isolation (detailed in `07-tenant-isolation.md`) is the *storage* sub-aspect realizing the canonical model.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | One concern; the conceptual domain model is canonical, storage realizes it. | Keeps business meaning independent of storage choice; lets storage be out-of-scope for stateless tools/PoCs without losing the domain model. |
| 2 | Owns ID'd `ENTITY-###` definitions referenced everywhere. | A single source for "what the data is", parallel to `CAP-###`; gives Interfaces/UX/Security/Quality one shape to build against, preventing drift. |
| 3 | Domain tags fields; Security & Governance own the policies. | Facts (which fields are sensitive) live where the fields are defined; rules (how they're protected/retained) live where they're reasoned about. The tag is the join. |
| 4 | Contract-grade requires checkable invariants (`INV-###`). | Turns business rules into assertable tests, aligning with the testable-acceptance principle and making the Verified handoff mechanical. |
| 5 | A uniqueness / exactly-once `INV-###` must name its enforcement mechanism. | A test detects a violation *after* it happens; only a constraint/index/transaction *prevents* it. A messaging product's exactly-once mention delivery had a non-unique index and no constraint, so a concurrent re-parse could emit two notifications/emails — the invariant read as satisfied but wasn't enforced (failure-mode #18a). |
| 6 | An id's format — including UUID version — is owned here and honored downstream. | The id *version* can be load-bearing (a v7 id is time-ordered and may back an ordering invariant), yet "id type = UUID" hides it. Domain picking v7 while Interfaces validated v4 422'd every id route at runtime — layer drift the ID web missed because the version wasn't in the id contract (failure-mode #18b; cross-ref Interfaces 11.4). |
| 7 | A derived key referenced by an `INV-###` is a contract to specify once. | If the derivation (inputs, sort order, encoding, delimiter) is left implicit, two modules compute the "canonical" key differently and silently fail to converge. Specifying it once is the same owned-once discipline applied to a computed value. |
| 8 | `ENTITY-USER` is conditioned on user accounts/identity existing. | The clause was unconditional, so a no-account product (a client-only browser game) had to argue the entity's absence into Non-goals. The contract is "where identity exists, it's modeled once here" — not "every product has users in its domain". |
| 9 | Enforcement-mechanism examples include **structural / by-construction** mechanisms. | The original examples were relational-DB-flavored (constraint / unique index / transaction); a product on IndexedDB enforced exactly-once structurally (fixed key, single-record `put()`, transaction scope) and had to argue that into the bar's letter. The bar's intent — the store cannot represent the violation — is store-agnostic, so the examples now say so. |
