# Concern 11.3 — Architecture

> **One-line:** The logical structure of the system — what the components are, what they're responsible for, how they interact, and the patterns and decisions that hold it together.

## Purpose & Scope

Defines the **logical** design: component decomposition, responsibilities, boundaries, interactions, cross-cutting patterns, technology choices, and the cross-cutting decision record. It does **not** own **physical** deployment/runtime (Operations & Infrastructure realizes the component model).

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Typical default | Notes |
|---|---|---|---|
| Component decomposition & responsibilities | `component-decomposition-responsibilities` | always in | The structural map. Includes **in-process execution placement** (e.g. worker vs main thread) where it is load-bearing |
| Boundaries & isolation model | `boundaries-isolation-model` | in for multi-component systems | Trust/data/process boundaries |
| Component interactions / data flow | `component-interactions-data-flow` | in for multi-component systems | Sync/async, who calls whom |
| Cross-cutting patterns | `cross-cutting-patterns` | in for non-trivial systems | Auth, error handling, logging, config |
| Technology choices | `technology-choices` | always in | With rationale (→ ADRs) |
| ADR register | `adr-register` | always in | Cross-cutting decisions (`ADR-###`) |
| Logical deployment topology | `logical-deployment-topology` | in for distributed systems | *Logical* only; physical → Ops. **In-process** concurrency placement is *not* topology — it belongs to component decomposition/interactions, so scoping this key out does not scope that out |
| Scalability / resilience patterns | `scalability-resilience-patterns` | optional | Overlaps Performance & Scalability |

## Owned Contracts (referenced by stable ID)

1. **Component model** — `COMPONENT-###`: responsibility, owned interfaces, dependencies. **Referenced by** Interfaces (each endpoint/command attributed to a component), Delivery Process (slices map to components), Operations (deploy units), Observability (telemetry per component).
2. **ADR register** — `ADR-###`: system-wide/cross-cutting decisions with context · status · consequences. **Referenced by** any concern (and the failure-mode rationale). Concern-*local* decisions stay in that doc's Design Decisions section.
3. **Cross-cutting patterns** — `PATTERN-###` (e.g. auth propagation, error envelope, structured logging). **Referenced by** every implementing concern so behavior is uniform.

## Consumed (referenced, not owned)

- `CAP-###` (Product & Requirements) — components realize capabilities.
- `ENTITY-###` (Domain & Data) — components own data.
- Physical realization (Operations & Infrastructure).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Architecture |
|---|---|
| **Absent** | No architecture doc (acceptable only for the most trivial single-file tool). |
| **Sketch** | High-level boxes — main components and how they connect, as bullets/a diagram. |
| **Specified** | Components described with responsibilities; boundaries and interactions in prose; tech stack chosen; key patterns named. |
| **Contract-grade** | `COMPONENT-###` with responsibility + owned interfaces + dependencies; cross-cutting `PATTERN-###` specified concretely enough to implement uniformly; `ADR-###` for key choices; boundaries explicit. An implementer can place new code correctly and follow consistent patterns without guessing. |
| **Verified** | *(delivery methodology)* The built system matches the component model and patterns (e.g. via structure/fitness checks). |

## Contract-grade bar

Components (responsibility + owned interfaces + dependencies) + specified cross-cutting patterns + ADRs for key decisions + explicit boundaries.

## Worked Example (a planning SaaS)

`02-system-architecture.md` is this concern at ~*Specified*. To reach *Contract-grade*:
- **Components:** `COMPONENT-PLATFORM-SVC`, `COMPONENT-IDENTITY-SVC`, `COMPONENT-MODEL-STRUCTURE-SVC`, `COMPONENT-MODEL-DATA-SVC`, `COMPONENT-VIEW-SVC`, `COMPONENT-AUDIT-SVC`, `COMPONENT-SCHEDULER-SVC`, `COMPONENT-KONG`, `COMPONENT-TENANT-APP`, `COMPONENT-PLATFORM-ADMIN-APP` — each with responsibility + owned interfaces + dependencies.
- **ADRs:** `ADR-TENANT-SCHEMA-ISOLATION`, `ADR-KONG-DBLESS`, `ADR-GO-WORKSPACE`, `ADR-CHI-ROUTER`.
- **Patterns:** `PATTERN-JWT-AT-GATEWAY` (Kong validates JWT, propagates trusted headers), `PATTERN-PER-SERVICE-AUTHZ`.
- **Boundaries:** platform / tenant / model layers.
- Physical realization (Helm, namespaces, scaling) is **referenced** to Operations & Infrastructure, not restated here.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Architecture owns the *logical* component model; Operations owns *physical* deployment. | Keeps design independent of deployment target; the same architecture can be realized in k8s, a VM, or serverless without rewriting the concern. |
| 2 | Owns an ID'd `COMPONENT-###` model referenced everywhere. | The structural spine (parallel to `CAP-###`/`ENTITY-###`): lets interfaces, slices, and deploy units all attribute to a component, enabling end-to-end traceability. |
| 3 | A dedicated cross-cutting `ADR-###` register complements per-doc Design Decisions. | System-wide decisions span concerns and need one authoritative home with context/status/consequences; concern-local decisions stay local. Two scopes, no duplication. |
| 4 | Contract-grade requires concretely specified cross-cutting patterns. | Uniform patterns are what keep an AI (or a team) from implementing auth/errors/logging five different ways; naming them isn't enough — they must be implementable as written. |
| 5 | A cross-cutting **error-handling pattern must be *total*** — it specifies how errors from **every** source are handled, including the framework/runtime (input validation/parsing), not only application-raised ones — **and content-negotiated** (HTML vs programmatic clients). For a **client-only** product with a single presentation surface, the negotiation half degenerates and the operative bar is **total + never console-only**: every error from every source reaches a user-visible surface, not just the developer console. | The common leak is a framework's *default* error response (e.g. a raw JSON 422) escaping the uniform envelope straight to the user. A pattern that only catches app-raised errors isn't actually uniform; "total + content-negotiated" closes the bypass. Realizes the Interfaces (11.4) total error-model rule. The client-only analogue exists because a browser product's equivalent bypass is an exception dying in the console while the UI sits silent — a validation build (a client-only browser game) had to derive this by analogy, so the bar now states it. |
| 6 | In-process concurrency placement (worker vs main thread) is **component decomposition**, not deployment topology. | A client-only validation build scoped out `logical-deployment-topology` (one process, no distribution) and then needed defensive wording to keep worker-vs-main-thread placement in scope — but that placement *is* load-bearing structure (it decides interaction contracts and message seams), so it lives with decomposition/interactions regardless of topology's scope. |
