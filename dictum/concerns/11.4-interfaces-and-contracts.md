# Concern 11.4 — Interfaces & Contracts

> **One-line:** Every surface through which the product is invoked, specified as a contract precise enough that both sides build to it without guessing. The single source the delivery methodology's work items reference.

## Purpose & Scope

Owns the canonical contracts for each active interface kind. One concern, **sub-typed by trait**: a product activates the sub-types it exposes (HTTP API, CLI, library/SDK, events, UI entrypoints) — often several at once. This is the primary anti-drift mechanism in the whole system.

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Activated when | Notes |
|---|---|---|---|
| HTTP/RPC API surface | `http-rpc-api-surface` | exposes a network API | `API-###` |
| CLI surface | `cli-surface` | has a command-line interface | `CLI-###` (commands, flags, exit codes) |
| Library / SDK surface | `library-sdk-surface` | consumed as a library | `LIB-###` (public functions/types) |
| Event surface | `event-surface` | publishes/consumes events | `EVT-###` (event schemas, topics). **Intra-process/worker message protocols** (e.g. worker `postMessage` payloads) are wire schemas too — suggested prefix `MSG-###`, on the same footing as `EVT-###` |
| UI entrypoints | `ui-entrypoints` | has a UI | The **navigation contract** — owned by User Experience (11.9), referenced here. Interfaces still owns the contract layer **below** screens: input bindings (key/pointer → action) and the stable DOM hooks E2E/a11y tests select against — suggested id schemes `BIND-###` / `DOM-###` |
| Error model / catalog | `error-model-catalog` | always (for in-scope kinds) | Shared `ERR-###` envelope + per-element errors |
| Versioning & compatibility | `versioning-compatibility` | scales with exposure | Detailed for public/library; light/out for internal & PoC |
| Pagination / filtering / rate-limit conventions | `pagination-filtering-rate-limit-conventions` | optional (APIs) | May reference Performance / Ops |

## Owned Contracts (referenced by stable ID)

Each programmatic element (`API-### / CLI-### / LIB-### / EVT-###`) carries:
`signature · typed inputs/outputs (ref ENTITY-###) · error catalog · pre/postconditions · side-effects/idempotency · owning COMPONENT-### · served CAP-###`.

A **response-projection** rule applies to every in-scope element:
- **Name the wire shape, not just the entity.** The typed output must be a **concrete response projection** — the fields exposed and their names (the wire DTO a client builds against) — **not merely the owning `ENTITY-###`**. "Returns `ENTITY-X`" is *not* build-ready: it leaves snake_case-vs-camelCase, which fields are exposed, and which are withheld (e.g. is `email` included?) for each layer to invent — exactly the drift the ID web exists to prevent, one level below the ID. (failure-mode #19.)

Two **error-model** rules apply to every in-scope element (the `ERR-###` envelope is shared, so they're stated once and bind all elements):
- **Inputs specify optionality + empty-value semantics.** For each parameter: required/optional, and **what an empty value means** (e.g. a blank/“all” filter = no filter, *not* a parse error). The value a UI control submits when unset (typically `""`) is part of the contract.
- **The error model is total and content-negotiated.** *Every* error — including errors raised by the framework/runtime (input validation/parsing), not only those the application raises — maps to the uniform `ERR-###` model and is **content-negotiated**: an HTML client receives the HTML error rendering; a programmatic client receives the structured error. **A raw framework error reaching the caller is a contract violation** (a contract test asserts it). *Client-only adaptation:* where no programmatic caller exists (a single in-browser surface), the negotiation half degenerates and the operative rule is **total + never console-only** — every error from every source maps to `ERR-###` and reaches a user-visible surface, never only the developer console (cross-ref Architecture 11.3 DD #5).

These per-element rules — typed I/O **with a concrete response projection**, error catalog, empty-value semantics, and the authz reference — **apply equally to the `event-surface` sub-type (`EVT-###`, and `MSG-###` intra-process message protocols) where present**, not only to the request/response surface. An event element specifies its payload projection, its error/failure behavior, and its **subscription/connection authorization** (handshake auth + per-resource subscribe checks; cross-ref Security 11.7) on the same footing as an `API-###`. (failure-mode #17.)

**Referenced by** Delivery Process (a slice's work item links the IDs it touches — see the delivery methodology's "enumerate the API contract in the work item"), User Experience (which calls which `API-###`), Quality & Testing (contract/integration tests), Integrations (external-facing contracts).

## Consumed (referenced, not owned)

- `ENTITY-###` (Domain & Data) — I/O shapes project from entities.
- `COMPONENT-###` (Architecture) — each element is owned by a component.
- `CAP-###` (Product & Requirements) — each element serves a capability.
- Security & Privacy — the authz applied to each element.
- UX navigation contract — for UI route entrypoints.

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Interfaces & Contracts |
|---|---|
| **Absent** | No interface doc. |
| **Sketch** | A list of endpoints/commands/functions by name. |
| **Specified** | Each element with a description and rough I/O, in prose. |
| **Contract-grade** | Full per-element spec: signature + typed I/O (ref `ENTITY-###`) **as a concrete response projection — the fields and their wire names, not just the owning entity** + **parameter optionality/empty-value semantics** + error catalog (**incl. input-validation, mapped to the total, content-negotiated `ERR-###` model**) + pre/postconditions + side-effects, with owning `COMPONENT-###` and served `CAP-###`; **validators honor the owned id format (incl. UUID version) from Domain 11.2**; **these rules apply equally to in-scope `EVT-###` event elements, incl. subscription authz**; versioning specified where the interface is exposed. Both sides can build, and a contract test can call it, without guessing. |
| **Verified** | *(delivery methodology)* Built interfaces conform; contract/integration tests pass. |

## Contract-grade bar

Every in-scope interface element fully specified (signature + typed I/O + parameter optionality/empty-value semantics + error catalog + pre/postconditions + side-effects + owning component + served capability); the `ERR-###` model is **total** (covers framework/validation errors) and **content-negotiated**; versioning detailed where exposed. Five additions, each strengthening an existing rule:

- **Concrete response projection, not just the entity.** Each element's typed output names the fields and their wire names exposed to the client — the DTO it builds against. "Returns `ENTITY-X`" is not build-ready; it forces each layer to invent the shape (failure-mode #19, which recurred across three implementers).
- **Validators honor the owned id format.** Input validation for an identifier must match the format Domain 11.2 owns, **including the UUID version** where it carries semantics (a v4-default validator rejecting v7 ids is a contract violation; failure-mode #18b).
- **Event-surface parity.** The bar's request-surface rules — typed I/O + concrete projection, error catalog, empty-value semantics, authz reference — **apply equally to the `event-surface` (`EVT-###`) where present**, including subscription/connection authorization (cross-ref Security 11.7; failure-mode #17).
- **One canonical bound.** A parameter/collection bound is stated on **one canonical quantity** and the other derived (e.g. state the size limit on *total* members, derive the *others* bound), so the DTO bound and the owning `INV-###` (Domain 11.2) cannot disagree.
- **A forced error names a forcing the proving tier can use.** Where an error-catalog row records how its condition is forced under test (a *Forced-by* cell) **and** a verification tier is assigned to it, the two are **authored together**: the named forcing mechanism must be usable at the tier that proves the row (a row owed an end-to-end "forced condition → user-visible surface" proof cannot list only unit-level forcings such as stubs and fake timers). Otherwise the tier/forcing mismatch surfaces only at build time, as a mid-slice doc edit. (failure-mode #23.)

## Worked Example (a planning SaaS)

`09-api-design.md` is the cautionary case: it declared the contract "auto-generated from code annotations" — i.e. *no canonical contract* — which is precisely how the two sides drifted. At *Contract-grade* it becomes:
- `API-VIEW-SAVE` — `PATCH .../views/{viewId}/data`; request = changed cells (ref `ENTITY-CELL`); response = applied diff; errors include per-cell `403`; side-effect: writes; owner `COMPONENT-VIEW-SVC`; serves `CAP-VIEW-GRID`.
- `API-MODEL-CREATE`, `API-CALC-RUN`, `API-DATA-EXPORT`, … each fully specified.
- **Versioning:** `/v1/` path versioning; internal-only, so the compatibility sub-aspect stays light.
- A slice's work item then lists, e.g., `touches: [API-VIEW-SAVE]` — referencing, not restating.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Kind-prefixed IDs (`API/CLI/LIB/EVT-###`) with full cross-links. | Makes every surface element a referenceable contract linking up to capability/component and down to entity shapes — end-to-end traceability across layers. |
| 2 | Interfaces OWNS the contracts; work items REFERENCE them by ID. | Single source of truth is the core anti-drift mechanism; it directly fixes the failure where code was the only contract and the two sides diverged. |
| 3 | Versioning scope scales with exposure. | You pay for compatibility rigor only where external consumers can break; internal interfaces and PoCs change freely (the scope lever). |
| 4 | Contract-grade demands full semantics, not just shapes. | Pre/postconditions and side-effects are where "looks the same, behaves differently" bugs hide; shape-only contracts let half-built behavior pass. |
| 5 | A concrete response projection is required, not just the owning `ENTITY-###`. | "Returns `ENTITY-X`" is one level above build-ready: it leaves field selection and wire naming to each layer, so independently-built clients drift below the ID. A messaging product's workspace/member/invitation/search surfaces forced three implementers to invent the JSON shape, while surfaces that gave an explicit DTO built cleanly (failure-mode #19). |
| 6 | The `EVT-###` event surface gets the same Contract-grade rules as the request surface. | The sub-type already exists, but the bar enumerated request-surface obligations and forgot events; an `EVT-###` with unspecified payload, errors, or subscription authz is as drift-prone as an unspecified `API-###`. Subscription authz in particular guards tenant isolation on the live path (failure-mode #17). |
| 7 | A bound is stated on one canonical quantity and the other derived. | Stating the same limit independently in the DTO and the invariant (on different quantities) is where an off-by-one ships; deriving one from the other removes the reconcile-by-arithmetic trap. |
| 8 | Interfaces owns the **DOM-hook + input-binding layer below screens** for UI products (`BIND-###` / `DOM-###`). | UX owns screens and navigation, but E2E and a11y verification need Interfaces-grade contracts one level down — which key/pointer input maps to which action, and which stable DOM selectors tests bind to. A client-only browser game (canvas + DOM) had to invent both prefixes mid-build because the split below `ui-entrypoints` was unnamed. |
| 9 | Intra-process/worker message protocols are first-class wire schemas (`MSG-###`), on the same footing as `EVT-###`. | A worker `postMessage` payload is a cross-context contract with all the drift potential of a network event, but it fit no published prefix — a validation build minted `MSG-###` locally. Naming it keeps every wire schema, in-process or not, under the same per-element rules. |
| 10 | A forced error row and its proving tier are authored together. | A catalog row gated at E2E carried only unit-level forcings (stub worker, fake timers); no end-to-end mechanism existed to force the condition, so the row's proof was unbuildable as written and the cell had to be re-authored mid-slice. The forcing column is part of the row's verification contract, not decoration (failure-mode #23). |
