# Concern 11.5 — Quality & Testing

> **One-line:** What "tested" means for this product — the test strategy, the coverage map over capabilities and invariants, and the real-flow E2E standard. Defines; Delivery Process enforces.

## Purpose & Scope

Owns the **test strategy**, **quality bars**, the **coverage map**, and the **real-flow E2E standard**. It does **not** own the Definition of Done, vertical-slice rule, or build sequence (Delivery Process) — Delivery *references* this concern's E2E standard for its proof-of-done gate.

## Owned Contracts (referenced by stable ID)

1. **Coverage map** — maps **every in-scope contract ID** to the test(s) that cover it, across the interface/domain/product set — not just the request/REST contracts. Concretely: each in-scope `CAP-###` maps to its required test types (real-flow E2E for user-facing); each `INV-###` maps to an assertion; each `API-###` to a contract/integration test; each **`EVT-###`** on the event surface to a test of its **delivery, authorization, and fan-out** behavior (a distinct contract no `API-###` test covers — the live/push path is exercised, not inferred from the request path); and each product **`SUCCESS-###`** to the test that demonstrates it. Prefer **one `INV-###` per test** (or a separately-identifiable assertion per invariant) so a failure attributes to a single invariant — the drift-detector's semantic check (Part 10d) relies on this. **Edge inputs, not only happy values:** every in-scope **input-bearing interface element** — an `API-###` parameter, a `CLI-###` flag, a `LIB-###` parameter, a form field — is tested at its **default and boundary inputs**: every optional input *absent* **and** *empty* (the value a UI submits for a blank/“all” control), plus one malformed value — asserting the contracted outcome (a valid result or the uniform error model, never a raw framework error). **An ID with no observable check gets an explicit `n/a` row stating why.** Some in-scope IDs are constitutionally untestable (a `PERSONA-###`, an advisory environment row, a factual boundary statement, an attestation); the map records the disposition rather than inviting a theater check — **silent omission is the defect**, the stated `n/a` is not. **Referenced by** Delivery Process (DoD checks coverage). The test-side complement to the traceability spine.
2. **Real-flow E2E standard** — `E2E-STANDARD`: a test exercises the system's **own** code paths for real (real authentication — no bypass / no token injection, real navigation, real backend logic) and asserts the real rendered outcome. **“Real navigation” means operating the screens, not merely reaching them:** the E2E **drives every interactive control** on each in-scope `SCREEN-###` it visits — every form/button (filter, run, search, sort, paginate, submit) — **at least once, including in its default/empty state**, and asserts a **non-error** rendered outcome (a real user clicks those buttons; the test must too). **External dependencies may be substituted by approved fakes per the environment's fidelity map** (owned by Operations): substituting an external (e.g. S3 → a container) is legitimate; bypassing own code (e.g. login) never is. The bar is thus *environment-relative*. Three further rules of the standard: **(a) expectation pre-computation via pure product modules is sanctioned** — a test may import the product's *pure* modules into the test process to compute expected values (e.g. deriving expected terrain or a feasible route from the product's own deterministic generator); this injects nothing into the system under test and is observation-adjacent, not a bypass. **(b) Determinism journeys assert run-invariant observables** — a journey claiming reproducibility must compare values that are identical across runs *by contract* (exact initial state, settled end-states, exact key/delta sets), never time-sampled trajectories, which differ run-to-run with frame timing even in a fully deterministic product. **(c) Policy-liveness checks bind to page-governed observables** — a check that a runtime-governed policy (e.g. a CSP) is *live* must observe what the **page** governs (injected-script non-execution, rejected requests, real violation events), never harness-context evaluation: automation harnesses run privileged and are typically exempt from the page's policy, so a harness-eval probe passes vacuously. **Referenced by** the delivery methodology's proof-of-done.
3. **Quality bars** — coverage thresholds, required test types, and gate definitions. Includes the **flake / re-run policy**: what a red run obligates (investigate; re-run scope), and whether retries/quarantine exist — recommended **zero retries, no quarantine list** for functional tiers; for *measurement* tiers the re-run rule is **co-defined with Performance & Scalability (11.13)**. The post-investigation disposition for a genuine transient (re-run green, zero code change) is **record the incident plus both runs** — never a quarantine list; a wall-clock-sensitive observation row may state an environment load-condition precondition instead.

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Quality & Testing |
|---|---|
| **Absent** | No test strategy. |
| **Sketch** | Which test types will be used, as bullets. |
| **Specified** | Test approach per type described; rough coverage goals; test-data approach noted. |
| **Contract-grade** | Coverage map over **every in-scope contract ID** — every `CAP-###` (incl. real-flow E2E for user-facing) + an assertion per `INV-###` + **edge/default inputs per input-bearing interface element** + **delivery/authz/fan-out per `EVT-###`** on the event surface + a demonstration per `SUCCESS-###`, with an explicit `n/a — why` row for any ID with no observable check; quality bars + required gates stated (incl. the flake/re-run policy); test-data strategy defined; `E2E-STANDARD` specified, **including operation of every in-scope screen's interactive controls (default state included)**. A team or AI can implement the tests without guessing what "tested" means. |
| **Verified** | *(delivery methodology)* Tests implemented and passing in CI. |

## Contract-grade bar

Coverage defined over **every in-scope contract ID** — per `CAP-###` (incl. real-flow E2E for user-facing) + an assertion per `INV-###` + **edge/default inputs per input-bearing interface element** (API param, CLI flag, LIB param, form field) + **delivery/authz/fan-out per `EVT-###`** (where an event surface exists) + a demonstration per `SUCCESS-###`, with an **explicit `n/a — why` row** for any in-scope ID that admits no observable check (silent omission fails the bar) — plus quality bars + a named **flake/re-run policy** (co-defined with 11.13 for measurement tiers) + test-data strategy + an `E2E-STANDARD` that **operates every in-scope screen's controls in their default state** (not just reaches the screens).

## Worked Example (a planning SaaS)

The CI conventions (`go test -race -count=1`, integration tags, Vitest) plus the **bypassed-login E2E** are this concern below *Contract-grade*. At *Contract-grade*:
- **`E2E-STANDARD`:** the E2E logs in through the real Keycloak flow as `PERSONA-MODEL-CONTRIBUTOR`, navigates the real sidebar to `CAP-VIEW-GRID`, saves a cell via `API-VIEW-SAVE`, and asserts the value renders on reload — **no auth bypass**.
- **Coverage map:** `CAP-VIEW-GRID` → unit (cell validation) + integration (view-svc + DB) + contract (`API-VIEW-SAVE`) + real-flow E2E.
- **Invariant assertion:** `INV-CALC-DAG` → a test that rejects a cyclic calculation.
- **Quality bars:** race detector on, integration-tagged suite required, frontend Vitest.
- Delivery Process's Definition of Done then *references* `E2E-STANDARD` rather than restating it.

A planning SaaS has no event surface, so its coverage map is `CAP/INV/API`-shaped. For a product that *does* expose one (e.g. a real-time messaging product), the same map adds a row per `EVT-###` — e.g. `EVT-S-MESSAGE-NEW` → a test of **delivery** (the message reaches subscribed clients), **authorization** (a non-member subscribing is rejected — failure-mode #17), and **fan-out** (every member of the conversation receives it) — behavior no `API-###` contract test exercises.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Quality owns the test strategy/bars; Delivery owns slicing/DoD and uses tests as proof. | Reusable "define vs enforce" split: the test strategy is a stable artifact; the DoD is a process gate that points at it. |
| 2 | Owns a coverage map over **every in-scope contract ID** — `CAP/INV/API` **and the event surface `EVT-###` + product `SUCCESS-###`**, not just request/REST contracts (failure-mode #10, generalized by #17). | Makes "is this tested?" objective and gap-visible, completing the traceability web from the test side. An event-driven product's real-time delivery/authz/fan-out is a distinct contract no `API-###` test covers; enumerating only `CAP/INV/API` would let that whole surface pass the bar untested. |
| 3 | Quality owns the real-flow E2E standard; Delivery references it. | One authoritative definition of "real flow" prevents the two-copies drift, and directly encodes the fix for the bypassed-login failure. |
| 4 | Contract-grade requires real-flow E2E for user-facing capabilities + an assertion per invariant. | These are the two coverage facts that most reliably catch the half-built and rule-violation failures; making them mandatory at Contract-grade ties testing to the system's core failure modes. |
| 5 | Real-flow means *own code real; external deps substitutable* per Operations' fidelity map. | Separates legitimate dev-stage substitution (mock S3) from illegitimate bypass (skip login), letting a developer be Done against local/mocked infra without weakening the anti-drift guarantee. |
| 6 | Edge-input coverage binds per **input-bearing interface element**, not per `API-###` only. | The clause was phrased per-`API-###`; a client-only browser game (UI + library surfaces, zero APIs) had to adapt it ad hoc to form fields and `LIB-###` params. The obligation was always about inputs, so the rule now names the element class instead of one prefix. |
| 7 | An in-scope ID with no observable check gets an explicit `n/a — why` row; silent omission is the defect. | Constitutionally untestable IDs (personas, advisory env rows, factual boundaries, attestations) made the literal "every ID maps to a test" bar unreachable — or worse, invited theater checks. The stated disposition keeps the map total without faking coverage. |
| 8 | Quality owns the flake/re-run policy; measurement-tier re-runs are co-defined with 11.13; an investigated transient is recorded (incident + both runs), never quarantined. | A validation build's zero-retry policy met a real single-occurrence wall-clock flake: the policy gave the right verdict (red → investigate → re-run whole) but named no owner and no post-investigation disposition, so the record-keeping was improvised. Quarantine lists rot; recorded incidents don't. |
| 9 | Expectation pre-computation via **pure product modules in the test process** is sanctioned observation-adjacent use. | E2E route/value planning sometimes needs the product's own deterministic facts (where terrain is, what a generator emits). Importing pure modules into the test process injects nothing into the system under test — without this rule every such product re-derives the legitimacy argument. |
| 10 | Determinism journeys assert **run-invariant observables**. | A prescribed "fixed probe walk" determinism check failed spuriously: real-time sampling is frame-timing-dependent, so identical worlds recorded different trajectories. Reproducibility claims must compare values that are identical across runs by contract. |
| 11 | Policy-liveness checks bind to **page-governed observables**, never harness-eval. | A CSP probe evaluated from the automation harness "passed" vacuously — harness contexts run privileged and are exempt from the page's policy. Only what the page itself governs (script non-execution, rejected requests, violation events) proves the policy live. |
