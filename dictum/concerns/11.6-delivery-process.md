# Concern 11.6 — Delivery Process

> **One-line:** How work is sliced, what "done" means, in what order it's built, and how a build-ready spec becomes a verified product. The bridge from the doc system to the running software.

## Purpose & Scope

Owns the **process spine**: the vertical-slice rule, the Definition of Done, the work-item hierarchy and slice-level designation, the build playbook, the build-ready gate scope, and **implementation-status (Verified) tracking**. It owns the `Verified` rung that every other concern hands off to. The slice/DoD/playbook rules are a **near-universal reusable core** (shipped as a template); only the hierarchy mapping, slice list, and gate scope are per-product.

## Owned Contracts (referenced by stable ID)

1. **Slice rule + fidelity-staged Definition of Done** — the reusable template (≈ `docs/00-delivery-methodology.md`). The DoD has **two gates** referencing Operations' environment fidelity map: a **merge gate** (developer stage — own code real, external deps substitutable per the map; the bar for code entering the repo) and a **release gate** (real-infra, owned by DevOps). Real-infra checks not runnable at merge are deferred to the release gate and **tracked, never silently skipped**. *Degenerate case:* for a product with **no substitutable externals**, the substitution half of the staged DoD is **stated as empty, not omitted** — the record carries an explicit "no externals; nothing substituted" so the gate stays checkable rather than reading as if something must be substituted.

   **Slice ≠ a `CAP-###` bijection.** Most slices realize one capability, but a slice is a vertical cut that ends in a verified increment, **not** a per-`CAP-###` row. A **cross-cutting / infrastructure slice** is a first-class case: a real `slice:full` that lights up a **shared path** used by several capabilities — e.g. the live-delivery / error-envelope / auth layer — verified end-to-end on its own, with the per-capability slices that ride that path landing after it. Such a slice maps to *several* `CAP-###` (or to none directly, when it's pure shared infrastructure), and is tracked in the build-status record like any other. A **verification-only slice** — one whose increment is *proof* (a test suite, an audit pass, a hardening battery) over an already-built surface — is likewise first-class. Do **not** force a strict one-slice-per-`CAP-###` bijection; do **not** invent a spurious `SLICE-###` ID web for it (slices are process artifacts, not ID-prefixed contracts). The playbook's "no deferred frontend/E2E" rule is read at **capability granularity**: no `CAP-###` completes without its real-flow E2E, but headless foundation slices (`slice:headless` layers a later journey rides on) may legitimately precede the first UI journey.
2. **Work-item hierarchy declaration** — which level is the slice level in this product's tracker (≈ `docs/00a-work-item-hierarchy.md`).
3. **Build-ready gate scope** — the set of in-scope `CAP-###` / concerns / sub-aspects whose Contract-grade triggers "build-ready".
4. **Build-status record** — per `CAP-###`/slice: built & verified, and **at which fidelity stage** (merge / release). **Kept separate from the manifest's documentation rung** (doc maturity ≠ implementation status).
5. **Tracker-binding declaration (optional)** — present only when work items live in an external tracker. A thin, product-local **adapter** — implementing the *execution-mirror* boundary — declaring **(a)** the *ID-carrying channel* — where a repo contract ID rides on a tracker item (a field / label / link) so the item *references* the contract, never restates it; **(b)** the *projection direction* — the derived backlog (Std Part 10e) and the build-status record above drive item creation + status **repo→tracker only**; **(c)** the *reconciliation rule* — **repo-wins** (a divergent tracker edit is a convenience, not a source). **Three item roles** (transitions of one item, not separate objects — the role is fixed by whether the item yet references a Contract-grade repo ID): **demand** — a request for new/changed behavior (business intent the repo doesn't own; ingested via `doc-feature`); **triage** — an unadjudicated defect hypothesis (repo-invisible until reproduced; then an externally-sourced `drift`-type change event, Std Part 10d); **execution** — a slice referencing contract IDs (the mirrored, buildable role). **Two gates:** *build* — no item is built until it references a repo contract ID at Contract-grade; *write* — a suspected defect earns its first repo write only on reproduction + adjudication (a code-defect verdict revokes `Verified` via a regression case and leaves the contract text untouched). **Boundary invariant:** the repo owns contracts and `Verified`/build-status; the tracker mirrors those (repo→tracker) and may additionally hold upstream demand and triage hypotheses. The repo **never** stores a tracker item ID/URL (a stored external edge would drift — Std Part 0.5); the only join is the stable ID, resolved at projection time.

## Consumed (referenced, not owned)

- `E2E-STANDARD` (Quality & Testing) — the proof-of-done gate references it.
- Environment **fidelity map** (Operations & Infrastructure) — the staged DoD's merge/release gates reference it.
- `API-###` etc. (Interfaces) — a slice's work item references the contracts it touches.
- `CAP-###` (slices realize capabilities), `COMPONENT-###` (slices map to components).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Delivery Process |
|---|---|
| **Absent** | No defined process. |
| **Sketch** | "We'll build in slices with tests", as bullets. |
| **Specified** | Slice rule, DoD, and sequence described in prose; hierarchy noted. |
| **Contract-grade** | Slice rule + full DoD + declared hierarchy/slice-level + build playbook + build-ready gate scope + Verified-tracking method, all concrete. An implementer can run the build with no process guesswork. |
| **Verified** | *Reflexive:* the process was actually followed and produced a verified product. (This concern *defines* Verified for all others.) |

## Contract-grade bar

Slice rule + full DoD + declared hierarchy/slice-level + build playbook + build-ready gate scope + Verified tracking — all concrete; and, when an external tracker is used, the tracker-binding declaration (channel + roles + gates).

## Worked Example (a planning SaaS)

`docs/00-delivery-methodology.md` + `docs/00a-work-item-hierarchy.md` are this concern at ~*Contract-grade*:
- **Reusable core:** vertical-slice rule (incl. the cross-cutting/infrastructure-slice case), `slice:full`/`slice:headless`, the DoD, the build playbook with anti-patterns. *(The cross-cutting slice arises when, e.g., "real-time delivery" lights up the live path for several `CAP-###` as one `slice:full`.)*
- **Per-product:** hierarchy declared as GitHub epic→sub-issue with the **slice level = sub-issue**; the slice list drawn from `CAP-###`; the build-ready gate scoped to the v1 (local-k8s-testable) capabilities.
- **Build-status record:** tracks which `CAP-###` slices are built and pass their real-flow E2E (`E2E-STANDARD`, referenced from Quality).

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | The delivery methodology is this concern's reusable core; products specialize the hierarchy/slice-list/scope. | Most of the methodology is universal; re-deriving it per product wastes effort and re-opens settled failure-mode fixes. A strong template ships with the standard. |
| 2 | Owns the process spine; references E2E standard, contracts, and IDs. | Process-owns-process; everything else by reference keeps the single-source-of-truth discipline intact. |
| 3 | Implementation status tracked separately from doc rung. | Preserves the handoff line (doc maturity ≠ build status) while still closing the loop from build-ready docs to a proven product. |
| 4 | Contract-grade includes the build-ready gate scope. | Explicitly naming what's in-scope to build now is what turns the global "all in-scope Contract-grade" gate into an actionable, unambiguous trigger. |
| 5 | The DoD is fidelity-staged: a merge gate (dev, externals substitutable) + a release gate (real infra, DevOps). | Developers can't validate against infra they don't control; gating merge on production fidelity blocks them. Staging the DoD lets "only Done goes to repo" coexist with deferred-but-tracked real-infra validation, while the "own code real, externals substitutable" rule keeps it honest. |
| 6 | A cross-cutting / infrastructure slice is a first-class slice, not a per-`CAP-###` bijection. | A real-time product's "real-time delivery" is a real `slice:full` lighting up the live path for *many* capabilities but is itself no `CAP-###`; a strict one-slice-per-CAP reading has nowhere to put it. Blessing the shared-path slice keeps the vertical-slice rule honest without inventing a `SLICE-###` web. |
| 7 | A **verification-only slice** is named alongside full/headless/cross-cutting. | A client-only validation build ended with an a11y/error-hardening slice whose entire increment was proof over built surface; the slice-type vocabulary only knew full/headless, so its legitimacy had to be argued. Naming the kind keeps slice plans honest about proof work. |
| 8 | "No deferred frontend/E2E" binds at **capability granularity**; foundation slices may precede the first UI journey. | A literal whole-build reading forbids legitimate headless layering (domain kernel before the first screen exists). The failure the rule prevents is a *capability* declared done without its real-flow proof — so that is the granularity at which it is stated. |
| 9 | The staged DoD's empty substitution set is **stated, not omitted**. | For a zero-external product the substitution half of the DoD read as vacuous, and the record template read as if something *must* be substituted. An explicit "nothing substituted" keeps the gate checkable and distinguishes "considered and empty" from "forgotten". |
| 10 | An **in-repo record is a valid tracker** for the hierarchy declaration. | "Declare the tracker" presumed an external tracker exists; an autonomous single-agent build tracks slices in the repo itself. The contract is the declaration, not the tool. |
| 11 | An external tracker is a **downstream mirror of what Dictum owns**, keyed on the stable-ID web (items reference contract IDs, never restate; the repo stores no tracker back-reference; contracts + `Verified`/build-status stay repo-authoritative). One item plays three transitioning roles — **demand** (a request; ingested via `doc-feature`), **triage** (an unadjudicated defect hypothesis — repo-invisible until reproduced, then a `drift` event whose code-defect verdict revokes `Verified` without editing the contract), **execution** (a mirrored, buildable slice). Two gates: no build before a Contract-grade ID; no repo write before a reproduced adjudication. | Extends owned-once (Part 5) + no-stored-edges (Part 0.5) across the repo boundary. A ticket must not become a shadow contract (failure-mode #2); an unreproduced bug must not pollute the oracle ("smells become `[REVISIT]`, never enshrined"); two sources of "done" would diverge. A bug is a *falsification hypothesis against the oracle*, not a rival truth — so it moves build-status, not the contract. Live two-way sync stays `[FUTURE-SCOPE]` (ROADMAP). |
