# Concern 11.7 — Security & Privacy

> **One-line:** The protection mechanisms — who can authenticate, what they're authorized to do, how secrets and sensitive data are protected, and what the trust boundaries are. The *controls*, not the *rules*.

## Purpose & Scope

Owns protection **mechanisms**: authentication, authorization (`ROLE-###` + per-interface authz), encryption, secrets handling, threat model, trust boundaries. It does **not** own data-handling **policy** — what protection/retention a classified field *requires* is owned by Governance & Compliance. The three-way chain: **Domain tags** fields → **Governance** sets the policy → **Security** implements the mechanism.

## Sub-aspects (the scope vocabulary — risk-scaled)

| Sub-aspect | key | Default | Pulled in / deepened by |
|---|---|---|---|
| Trust boundaries | `trust-boundaries` | in | always (the Baseline minimum) |
| Secrets & credential handling | `secrets-credential-handling` | in | always |
| Authentication mechanism | `authentication-mechanism` | in when users exist | auth, sessions |
| Authorization: `ROLE-###` (tenant- & resource-scoped) + per-interface mapping | `authorization` | in when multi-role/protected | distinct roles, protected resources, an event/subscription surface |
| Threat model | `threat-model` | light by default | network exposure, real PII, handled secrets. For **minimal-risk** products its job inverts: it is where "not a threat here" becomes a **contractual negative assertion** (e.g. "zero network egress") — clarifying, not skippable |
| Encryption (at rest / in transit) | `encryption` | risk-scaled | sensitive data, regulated |
| Session management | `session-management` | when auth present | web/UI sessions |
| Data-protection mechanisms per sensitive field | `data-protection-mechanisms-per-sensitive-field` | when sensitive data exists | realizes Governance policy |

## Owned Contracts (referenced by stable ID)

1. **Role model** — `ROLE-###` → permissions. A role is **tenant-scoped** when it derives from a `PERSONA-###` (`PERSONA-### → permissions`), or **resource-scoped** when it derives from a **relationship/membership entity** — a role a subject holds only *with respect to a particular resource*, read off that entity's role field (e.g. `ROLE-CHANNEL-ADMIN` from `ENTITY-CONVERSATION-MEMBER.role`), with **no backing `PERSONA-###`**. Both kinds are `ROLE-###` and feed per-interface authz identically; the difference is only what the role is checked *against* (the tenant vs. the specific resource).
2. **Per-interface authorization** — for each `API-###`/`CAP-###`, the required `ROLE-###` and any finer checks. **Where a product exposes an event/subscription surface (`EVT-###`), authz covers it too:** handshake authentication on connect **plus a per-resource subscribe check** — an `EVT-C-SUBSCRIBE` to a resource (e.g. a conversation) is authorization-checked **exactly like the equivalent REST read** of that resource. An unchecked subscription bypasses tenant isolation on the **live path**, so the request-surface bar binds the event surface (failure-mode #17). **Referenced by** Interfaces and User Experience (which gates which nav/route). Directly addresses the "role checks never wired" failure.
3. **Trust boundaries & threat model**; **secrets-handling** approach. Boundary and hygiene assertions carry the suggested id scheme **`SEC-###`** (e.g. a zero-egress assertion, a storage-locus rule, a byte-asserted security policy in a shipped file) so they are referenceable and coverable like any contract — for a minimal-risk product, the *entire* Contract-grade surface of this concern may consist of such `SEC-###` negative/hygiene assertions, and the threat model is where "X is not a threat here" is stated contractually rather than left implicit.

## Consumed (referenced, not owned)

- `PERSONA-###` (Product & Requirements) → tenant-scoped roles.
- `ENTITY-###` membership/relationship entities (Domain & Data) → resource-scoped roles (read off the entity's role field).
- Classification tags (Domain & Data) → which fields need protecting.
- **Data-handling policy (Governance & Compliance)** → Security implements the mechanism realizing each policy.
- `API-###` **and `EVT-###`** (Interfaces) → authz is applied per element (request surface and event/subscription surface alike).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Security & Privacy |
|---|---|
| **Absent** | No security doc (only a truly trivial offline tool). |
| **Sketch** | Auth approach + trust boundaries noted (the Baseline minimum). |
| **Specified** | Authn/authz described; roles listed; secrets approach; threat areas noted. |
| **Contract-grade** | `ROLE-###` (tenant- **and** resource-scoped) + per-interface authz — **covering the event/subscription surface (`EVT-###`) where one exists** — + authn mechanism + secrets handling + a protection mechanism for each sensitive field (per Governance policy) + trust boundaries. ## Contract-grade bar

Roles (tenant- and resource-scoped) + permission model + authn + per-interface authz (request surface **and** the event/subscription surface where one exists) + secrets handling + protection mechanism per sensitive field + trust boundaries (scoped to the risk pulled in).

## Worked Example (a planning SaaS)

`04-authentication.md` + `07-tenant-isolation.md` + the role model are this concern — and `04` below *Contract-grade* (protocol-only, no journey, bypassable). At *Contract-grade*:
- **Roles:** `ROLE-PLATFORM-ADMIN`, `ROLE-TENANT-ADMIN`, `ROLE-MODEL-ADMIN`, `ROLE-MODEL-CONTRIBUTOR` (from `PERSONA-###`).
- **Authn:** Keycloak OIDC — the *mechanism/flow* contract (the login *screen* is UX; the real-flow E2E in Quality exercises it).
- **Per-interface authz:** `API-VIEW-SAVE` requires `ROLE-MODEL-CONTRIBUTOR` + per-cell check.
- **Trust boundary:** schema-per-tenant isolation.
- **Field protection:** `ENTITY-USER.email (pii)` → access-controlled/encrypted *per the Governance policy* for `pii`.

A planning SaaS's roles are all persona-derived and it has no event surface. A product with both (e.g. a real-time messaging product) extends the same model: a **resource-scoped role** `ROLE-CHANNEL-ADMIN` derived from `ENTITY-CONVERSATION-MEMBER.role` (no backing persona, checked against the specific conversation), and **event-surface authz** where `EVT-C-SUBSCRIBE` to a conversation is authz-checked exactly like the REST read of that conversation — an unchecked subscribe would bypass `INV-TENANT-ISOLATION` on the live path (failure-mode #17).

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Security owns mechanisms; Governance owns data-handling policy; Domain owns the tags. | A clean facts→policy→mechanism chain; the same `pii` tag drives a Governance rule and a Security control without either duplicating the other. |
| 2 | Owns `ROLE-###` + per-interface authz mapping, **covering the event/subscription surface (`EVT-###`) as well as `API-###`** (failure-mode #17). | Makes "who can call/see/**subscribe to** what" explicit and referenceable, so UX and Interfaces can wire guards — fixing the unguarded-path failure. An unchecked `EVT-C-SUBSCRIBE` bypasses tenant isolation on the live path just as an unguarded REST read would; the request-surface bar must bind the event surface too (verified in practice: a non-member was rejected on subscribe only once event-surface authz existed). |
| 3 | A `ROLE-###` may be **resource-scoped** (derived from a relationship/membership entity) as well as **tenant-scoped** (derived from a `PERSONA-###`). | Some real authorization facets have no backing persona — e.g. a chat product's `ROLE-CHANNEL-ADMIN` is read off `ENTITY-CONVERSATION-MEMBER.role` and means "admin *of this conversation*", not a tenant-wide persona. Both kinds remain plain `ROLE-###` and feed per-interface authz identically; the only difference is what the role is checked against (the tenant vs. the specific resource). Framing roles as persona-only would leave resource-scoped authz with no home in the ID web. |
| 4 | Baseline depth scales via risk-driven sub-aspect scope. | A trivial tool gets trust-boundaries + secrets; a high-risk product (real PII, network-exposed, handling secrets) gets the full threat model and per-field protection. Risk/scope mismatch warns. |
| 5 | Contract-grade requires a protection mechanism per sensitive field. | Ensures every classified field is actually protected, not just tagged — closing the gap between "we know it's PII" and "we did something about it." |
| 6 | Boundary/hygiene assertions get a published id hint: `SEC-###`. | A minimal-risk validation build (a client-only browser game) had its whole Security surface — zero-egress, storage locus, CSP bytes — living on a locally coined prefix because no published scheme covered assertions that are neither roles nor per-interface authz. Publishing the hint keeps such contracts in the ID web without per-product invention. |
| 7 | For minimal-risk products, the threat model's job is the **contractual negative assertion**. | The sub-aspect was framed purely risk-additive ("pulled in by exposure/PII/secrets"), so it read as skippable exactly where it is most clarifying: a product whose security story is "no egress, no accounts, no secrets" needs those *absences* stated as checkable contracts, and the threat model is their home. |
