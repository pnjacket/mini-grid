# Concern 11.8 — Governance & Compliance

> **One-line:** Are we *allowed*, and are we *accountable* — the data-handling policy, regulatory obligations, audit requirements, and licensing/IP. Owns the *rules*; Security implements the *controls*.

## Purpose & Scope

Owns **all data-handling policy** plus regulatory compliance, audit policy, retention/residency, and license/IP. Completes the chain: **Domain tags** facts → **Governance** sets policy → **Security** implements mechanism (and **Observability** implements audit). The same chain runs with **Business & Legal (11.15) as the facts source**: where 11.15 owns a legal/IP constraint (e.g. a trademark clean-room rule), the **enforcement policy** realizing it is a Governance-owned `POLICY-###` — the legal constraint plays the role Domain's tags play for data. It does **not** own protection mechanisms or logging implementations.

## Sub-aspects (the scope vocabulary — regulation-scaled)

| Sub-aspect | key | Default | Pulled in / deepened by |
|---|---|---|---|
| License & IP compliance | `license-ip-compliance` | in | always (the Baseline minimum) |
| Data-handling policy (`POLICY-###`) | `data-handling-policy` | in when sensitive data exists | per classification tag |
| Audit requirements | `audit-requirements` | in when accountability matters | regulated, multi-tenant |
| Compliance framework mapping | `compliance-framework-mapping` | light by default | GDPR / HIPAA / SOC2 / sector-specific regimes |
| Records retention & data residency | `records-retention-data-residency` | risk-scaled | regulated, multi-region |
| Consent management | `consent-management` | when personal data + consent basis | |

## Owned Contracts (referenced by stable ID)

1. **Compliance policy** — `POLICY-###`. The prefix covers **compliance policy generally**; the canonical case is *data-handling* policy keyed to Domain's classification tags (protection level, retention, residency, consent), but an IP-compliance, legal-constraint-enforcement (facts owned by 11.15), or other regulatory policy carries the same prefix — no local re-mint needed. **Referenced by** Security & Privacy (implements the mechanism) and Domain & Data (lifecycle realizes retention).
2. **Audit policy** — what must be auditable + audit-log retention. **Referenced by** Observability & Monitoring (implements the trail).
3. **Framework obligation → control map** — each applicable obligation mapped to an owning concern/control. **Referenced by** those concerns.
4. **License/IP register** — dependency licenses + ownership, cleared; **plus the product's own outbound license stance**, including a *deliberate absence* ("no license granted / all rights reserved") — the stance is a contract either way, not only the inbound column.

## Consumed (referenced, not owned)

- Classification tags + `ENTITY-###` lifecycle (Domain & Data).
- Security mechanisms (realize the protection policy); Observability (realizes the audit policy).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for Governance & Compliance |
|---|---|
| **Absent** | No governance doc (only a truly throwaway tool). |
| **Sketch** | License/IP basis noted; obvious obligations flagged (the Baseline minimum). |
| **Specified** | Data-handling policy described; applicable frameworks named; audit needs noted. |
| **Contract-grade** | `POLICY-###` per classification tag; framework obligations mapped to owning controls; audit requirements + retention/residency stated; license/IP cleared. Checkable: every tag has a policy, every obligation an owner. Scoped to whatever regulation pulled in. |
| **Verified** | *(delivery methodology)* Controls in place and evidenced. |

## Contract-grade bar

Policy per classification tag + framework obligations mapped to controls + audit requirements + retention/residency + license/IP cleared.

## Worked Example (a planning SaaS)

`10-audit-logging.md` + `14-licensing.md` + `15-eula.md` + the residency aspects of `07-tenant-isolation.md` are this concern. At *Contract-grade*:
- **`POLICY-PII`** (for `ENTITY-USER.email`): access-controlled, retention period, EU residency; Security implements, Domain's lifecycle enforces retention.
- **Audit policy:** model changes and security events must be auditable, retained N years — the audit-service (Observability) implements the trail.
- **Frameworks:** EU tenants → GDPR data-subject rights mapped to controls (export → `CAP-DATA-EXPORT`; erasure → soft/hard-delete lifecycle); SOC2 controls mapped.
- **License/IP:** Go/Angular dependency licenses cleared.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Owns `POLICY-###` keyed to classification tags. | Completes the facts→policy→mechanism chain with a referenceable spine; Security and Domain resolve "the rule for this field" by tag. |
| 2 | Owns audit *policy*; Observability owns the *mechanism*. | Same facts/policy/mechanism split as data protection — keeps obligations with Governance and implementation with the logging concern. |
| 3 | Minimal license/IP baseline always; framework mapping only when regulated. | Even hobby tools have OSS-license obligations; full framework mapping is wasted on the unregulated. The scope lever pays only for what applies. |
| 4 | Contract-grade maps every obligation to an owning control. | Turns "we must comply" into "here is who/what complies", making compliance buildable and auditable rather than aspirational. |
| 5 | `POLICY-###` is **compliance policy generally**, not data-handling policy only. | A validation build's IP-compliance policy had to reuse the prefix via the local-allowance escape hatch because the definition was keyed narrowly to classification tags. The facts→policy→mechanism chain is the same whatever the facts source, so the prefix matches its actual use. |
| 6 | Enforcement policies for **11.15-owned legal/IP constraints are Governance-owned** (legal-constraint-as-facts). | A trademark clean-room constraint (owned by Business & Legal) needed an enforcement policy; the chain worked but had to be hand-derived because only Domain-tags-as-facts was named. Naming the pattern makes the 11.15→11.8→mechanism chain as mechanical as the tag chain. |
| 7 | The register includes the product's **own outbound license stance, incl. deliberate absence**. | The vocabulary covered inbound dependency licenses only; a product that deliberately ships *without* granting a license had no contract shape for that decision. An unstated stance is indistinguishable from a forgotten one. |
