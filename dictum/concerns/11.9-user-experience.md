# Concern 11.9 — User Experience

> **One-line:** The screens, the navigation that connects them, the journeys through them, and the states they show — each screen wired to the capability it realizes, the interface it calls, and the role that guards it.

## Purpose & Scope

Active when the product has an interactive UI. Owns the **navigation contract** and **screen specifications**, binding each screen to its capability, interface, and guard. It does **not** own a11y/i18n (Concern 11.14, referenced) nor the design system itself (parked `[REVISIT]`).

## Sub-aspects (the scope vocabulary)

| Sub-aspect | key | Default (when UI present) | Notes |
|---|---|---|---|
| Navigation / IA contract | `navigation-ia-contract` | in | `SCREEN-###` (+ `ROUTE-###` **where a URL surface exists** — see below) + menu + guard + reachability |
| Screen specifications | `screen-specifications` | in | realizes `CAP`, calls `API`, gated by `ROLE` |
| States | `states` | in | **display** states (loading / empty / error, not just happy path) **and interaction/input states** — each interactive control's default/empty submission (a blank/“all” filter, an unset field) is a spec'd state. **For a live/event-driven UI, also the event-driven / ambient states** (see below) |
| User journeys | `user-journeys` | in | what the real-flow E2E traverses |
| Component reuse / design-system binding | `component-reuse-design-system-binding` | in when a component library exists (interim rule) | reuse shared library; `[REVISIT]` for real binding. **Defaults out when no library exists** (e.g. a zero-dependency hand-rolled UI) — what remains is the in-product visual-consistency rule, owned by the screen specs |
| Responsive / layout | `responsive-layout` | optional | |

### Event-driven / ambient states (live UIs)

The display/interaction/input states above are **pull-driven** — they answer "what does the screen show when the user clicks?" The ambient bar triggers on **any state change not produced by a user action**. A server-pushed event surface (Interfaces `EVT-###`) is one instance; **client-local asynchrony** — timers (an autosave firing), focus/lock changes (a pointer-lock loss pausing the app), async worker results, completion of a background load — produces exactly the same class of load-bearing states that no click produces, even in a product with zero server push. For any screen with such states, the Contract-grade bar **additionally** covers them per screen (or per the persistent shell, where they live globally). For the server-pushed instance, the canonical set is:

- **Connection state** — `connected` / `reconnecting` / `disconnected` / `backfill` (catching up after a reconnect). The UI must show which it is and behave sanely in each (e.g. queued vs dropped actions while disconnected).
- **Presence** — who is online/away/offline, where presence is surfaced.
- **Typing** — typing indicators (a high-churn ephemeral state).
- **Unread dividers / counts** — the "new since you left" boundary and unread markers, and how they resolve on read.
- **Server-pushed error frames** — a server-pushed error event (`EVT-S-ERROR`) is a first-class state, not a transport detail: the screen specifies how it surfaces (and it goes through the same total, content-negotiated error model — Interfaces 11.4 / Architecture 11.3 — not a raw frame).

For the client-local instance, the same discipline applies to whatever the product's non-user-action triggers are (e.g. timer-driven save toasts, a lock-loss pause takeover, worker-completion appearance of content) — enumerated as an **ambient-state register** when they live shell-globally.

This is the event-surface analogue of #16: #16 made the bar **operate every click-driven control**; a live UI's defects hide in the **push/ambient** states the user never clicks. (Failure-mode #17.)

### SPA-shell pattern (one persistent shell, layered screens)

A `SCREEN-###` need not be a distinct page. The **SPA-shell pattern** is blessed: one persistent shell (e.g. a 3-pane chat layout, where the connection/presence/ambient states above typically live) with most screens realized as **panels or modals layered over it**. Each such screen is still a `SCREEN-###` with the full contract — and is still **deep-linkable** and **reachability-checked**: no-orphans is judged by "every screen is reachable and every link resolves" exactly as for page-like screens, the route just resolves to a panel/modal over the shell rather than a full navigation.

`ROUTE-###` ↔ `SCREEN-###` is **N:1** (a screen may be reachable by several routes — a canonical path plus query-param / modal routes) and never N-screens-per-route. For a client-routed SPA, `ROUTE-###` names client routes (including query-param and modal routes), not only server paths; each `ROUTE-###` resolves to exactly one `SCREEN-###`.

### Routeless products (no URL surface)

`ROUTE-###` is **conditional on a URL surface existing**. A product whose screens are app states rather than URLs (e.g. a pointer-locked canvas game with title/playing/paused states) mints **no** `ROUTE-###` — that would be the spurious-prefix anti-pattern. Instead the navigation contract binds to an **owned state-machine register** (e.g. `NAV-###` transitions over the app-state set), and the no-orphans rule generalizes accordingly: **every screen is reachable via a spec'd transition, and every transition resolves to an existing screen** — the same reachability check, with transitions where routes would be. Deep-linkability applies only where URLs exist.

### The `layer` screen kind

Some surfaces are never *navigated to*: a debug overlay, a toast, a banner. These are a third screen kind — a **layer** — alongside page-like screens and SPA-shell panels/modals. A layer carries the screen contract like any other, but its reachability fact is its **toggle/trigger** (the input that shows it, or the ambient event that raises it); a purely ambient layer may equivalently be specified as rows of the ambient-state register above. Either way it is *somewhere* — an unclassified overlay is an orphan by another name.

### Modal integrity (bar tightening)

**A modal's inertness over the background must be input-channel-complete: pointer AND keyboard.** A scrim that intercepts clicks while Tab/focus traversal still reaches the background controls is **not a modal** — background interaction through *any* input channel is a correctness bug in this concern's bar, not an accessibility refinement. (It surfaced as exactly that: a Tab-bypass of a click-blocking scrim, caught only because a11y happened to be in scope — with 11.14 scoped out it would have shipped.) Accessibility (11.14) owns only the focus-order *refinement* within an already-inert modal (initial placement, wrap order, restore-on-close); the inertness itself is owned here.

## Owned Contracts (referenced by stable ID)

1. **Navigation contract** — `SCREEN-### / ROUTE-###` (or `SCREEN-###` + the state-machine register, for routeless products): each entry = route/spec'd transition + menu placement + guard — **the `ROLE-###` where an auth surface exists, else the screen's enabling condition** (e.g. "a valid save exists") — + reachable-from parent + realized `CAP-###` + **the interface surface it drives** (the called `API-###` where one exists; the `LIB-###`/input-binding surface for a client-only product) + required states + **its interactive controls** (each form/button + the interface element it submits to or drives + its default/empty-input state). **Referenced by** Delivery Process (the nav-wiring DoD item; slices), Quality & Testing (the real-flow E2E **operates** these controls — 11.5), Security (guards). Fixes orphan screens, dead links, **and "button does something weird" controls the E2E never clicked**.
2. **User journeys** — `JOURNEY-###`: the real path a `PERSONA-###` takes. **Referenced by** Quality's real-flow E2E.

## Consumed (referenced, not owned)

- `CAP-###`, `PERSONA-###` (Product & Requirements); `API-###` (Interfaces); `ROLE-###` (Security); `ENTITY-###` (Domain, for display).
- Accessibility & Internationalization (11.14); the design system (parked `[REVISIT]`).

## Maturity Ladder — what each rung means HERE

| Rung | Meaning for User Experience |
|---|---|
| **Absent** | No UI (a non-UI product — concern not in scope). |
| **Sketch** | Screen list + rough navigation, as bullets. |
| **Specified** | Screens described with content; journeys outlined; navigation in prose; states noted. |
| **Contract-grade** | `SCREEN-###` with route (or spec'd transition) + nav-wiring + guard (role or enabling condition) + realized `CAP-###` + the interface surface it drives + required states (loading/empty/error) + **enumerated interactive controls and their default/empty-input states** + **event-driven/ambient states for any non-user-action state change** (server push or client-local) + **input-channel-complete modal inertness** + journey placement; component-reuse rule applied where a library exists. An implementer builds each screen wired into real nav and its real interface surface, with all states and controls, no guessing. |
| **Verified** | *(delivery methodology)* Screens built, wired, and the real-flow E2E passes. |

## Contract-grade bar

Per screen: route (or spec'd transition where no URL surface exists) + nav-wiring + guard (`ROLE-###` where auth exists, else the enabling condition) + realized `CAP-###` + the interface surface it drives (`API-###` where one exists) + required states + **enumerated interactive controls with their default/empty-input states** + **the event-driven/ambient states for any state change not produced by a user action** (server-pushed connection / presence / typing / unread / `EVT-S-ERROR`, or client-local timer/worker/lock-change states) + journey placement; interim component-reuse rule applied where a component library exists. A `SCREEN-###` may be a panel/modal over a persistent SPA-shell — still deep-linkable, reachability-checked, and `N:1` with its `ROUTE-###` — or a **layer** whose reachability fact is its toggle/trigger. **Bar tightening:** every modal's background inertness is **input-channel-complete — pointer AND keyboard** — as part of *this* bar; a Tab-reachable background behind a click-blocking scrim fails Contract-grade UX regardless of whether Accessibility (11.14) is in scope.

## Worked Example (a planning SaaS)

`13-tenant-ux.md` / `12-platform-admin-ux.md` / `16-views.md` — where missing-frontend, dead-links, orphan-screens, and missing-states failures concentrated. At *Contract-grade*:
- **`SCREEN-GRID-VIEW`:** route `/models/:id/views/:viewId`; sidebar entry under the model; guarded by `ROLE-MODEL-CONTRIBUTOR`; realizes `CAP-VIEW-GRID`; calls `API-VIEW-SAVE`; states = loading grid / empty "no assets" / per-cell save error; in `JOURNEY-CONTRIB-ENTRY` (Login → Dashboard → Model → View List → Grid).
- **Navigation contract:** every screen reachable, every link resolves to an existing screen.
- **Interim rule:** reuse the shared Angular component library; no bespoke styles. `[REVISIT]` for the design-system binding.

## Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | UX owns a navigation *contract*, not navigation prose. | Making route + menu + guard + reachability a deliverable is what prevents orphan screens and dead links — the failure where nav was described but never built. |
| 2 | `SCREEN-###` binds `CAP` + **the interface surface it drives** + a guard (`ROLE` where auth exists, else the enabling condition). | The screen is the UI-side integration point; binding it to its backing surface + guard makes single-layer "done" impossible in either direction. Generalized from `CAP + API + ROLE`: a client-only product has no `API-###` and may have no auth surface, but the screen still drives *some* contracted surface (`LIB-###`/input bindings) and still has a stated guard — "enabling condition" keeps the slot filled instead of vacuous. |
| 3 | Interim component-reuse rule now; `[REVISIT]` for the design system. | Keeps the surface consistent without a consumable design system; the design-tool binding stays parked until one is in scope. |
| 4 | Contract-grade requires loading/empty/error states + journey placement. | These are exactly what was missing when features shipped happy-path-only; mandating them ties UX completeness to the real failure modes. |
| 5 | A live/event-driven UI must also spec event-driven/ambient states (connection/presence/typing/unread/`EVT-S-ERROR`). | #16 hardened the bar around *click-driven* controls; a live product's load-bearing states are **push/ambient** and the user never clicks them, so they'd pass the click-centric bar while absent (failure-mode #17). |
| 6 | Bless the SPA-shell pattern; `ROUTE-###`↔`SCREEN-###` is N:1. | A real SPA is one persistent shell with panel/modal screens layered over it; forcing "one screen = one page" fought the actual product. A panel is still a `SCREEN-###` with the full contract, still deep-linkable and reachability-checked — so no-orphans holds without page-shaped screens. |
| 7 | `ROUTE-###` is conditional on a URL surface; routeless products bind navigation to an owned state-machine register. | A pointer-locked canvas product has app states, not URLs — minting `ROUTE-###` there is the spurious-prefix anti-pattern. The reachability *intent* survives the generalization: every screen reachable via a spec'd transition, every transition resolving to a real screen. |
| 8 | The ambient bar triggers on **any non-user-action state change**; server push is one instance. | The v0.11 bar named only server-pushed `EVT-###` states, so a client-only product's timers, lock-loss pauses, and worker-completion states — the exact failure-mode-#17 class — read as out of the bar's reach. The trigger is the *provenance* of the state change (not a user action), not the transport (failure-mode #22). |
| 9 | A **layer** is a third screen kind; its reachability fact is its toggle/trigger. | Debug overlays, toasts, and banners fit neither page nor panel, leaving the no-orphans rule undefined for them — and an unclassified surface is an unchecked one. Tying reachability to the trigger keeps the register total. |
| 10 | **Bar tightening:** modal inertness is **input-channel-complete** (pointer AND keyboard); 11.14 owns only focus-order refinement. | A modal whose scrim blocked pointer input was Tab-bypassable; the defect surfaced only because a11y happened to be in scope. Background interaction through any input channel is a UX correctness bug — splitting it by channel had parked half the bug in an optional concern (failure-mode #20). |
| 11 | The component-reuse interim rule defaults out when no component library exists. | The rule presumed a shared library; for a zero-dependency hand-rolled UI it was meaningless as written. What survives is the in-product visual-consistency obligation, owned by the screen specs themselves. |
